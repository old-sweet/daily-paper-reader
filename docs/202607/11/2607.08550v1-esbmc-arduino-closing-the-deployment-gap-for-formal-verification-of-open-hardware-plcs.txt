# ABSTRACT 

OpenPLC, Arduino OPTA, CONTROLLINO, and Industrial Shields M-Duino – bring IEC 61131-3 programming to low-cost, microcontroller-class devices increasingly used in real automation and in Industrial Control Systems (ICS) security research. Existing open-source verifiers for IEC 61131-3, including the ESBMC-PLC line, prove safety over an abstract scan-cycle model with idealized unbounded integers. The artifact that runs on the board, however, executes on a resource-constrained Microcontroller Unit (MCU) with a 16-bit machine word (8-bit AVR Arduinos), and sensors are read through a finite-resolution Analog-to-Digital Converter (ADC). We show that this deployment gap makes naive width-aware verification unsound in practice : on 123 real third-party programs, checking 16-bit arithmetic overflow without a hardware input model raises a 44 % false-alarm rate (54/123) and finds zero genuine defects, because it explores sensor values no ADC can produce. Because the gap sits exactly where computation meets the physical process – a bounded sensor reading scaled by finite-width integer arithmetic into an actuation command – an overflow can silently suppress a physical safety action such as a high-level alarm. At the same time, an unbounded input model fabricates alarms that no environment can trigger. We present a method for hardware-faithful verification of IEC 61131-3 on open hardware : a declarative Hardware Abstraction Layer (HAL) descriptor (word width, ADC/Pulse-Width Modulation (PWM) resolution, I/O binding) and a sound lowering that interprets arithmetic at the target word width and constrains each input to its hardware-realizable range, transferring to the Programmable Logic Controller (PLC) domain the target-model discipline established for safety-critical embedded C (Frama-C, Astrée). We instantiate it for Arduino as ESBMC-Arduino , deriving HAL parameters from the official Arduino cores and realizing the input-range model as a lightweight hook in the Efficient SMT-based Context-Bounded Model Checker (ESBMC) Ladder Diagram (LD) frontend. On the 123-program corpus, the HAL annotator eliminates all 54 false alarms (54 →0) while preserving every robustness proof, and a controlled corpus demonstrates the genuine – but rare – width-dependent defects the method detects with physically realizable witnesses.

# 1 Introduction 

Programmable Logic Controllers (PLCs) govern safety-critical processes, but a new class of open-hardware PLCs – OpenPLC [ 1, 2 ], Arduino OPTA [ 3 , 4], CONTROLLINO [ 5], and Industrial Shields M-Duino [ 6] – now brings International Electrotechnical Commission (IEC) 61131-3 programming to low-cost, microcontroller-class devices. These platforms are widely used in education, small-scale automation, and – importantly for assurance – as the runtime substrate for academic Industrial Control Systems (ICS) security testbeds; OpenPLC, in particular, was designed explicitly for cybersecurity research [1]. 

1.1 The Deployment Gap 

Open-source verifiers for IEC 61131-3 – the ESBMC-PLC in particular [ 7 , 8, 9], and PLCverif [ 10 , 11 ] – prove safety properties over an abstract scan-cycle model of the Ladder Diagram (LD) or Structured Text (ST) logic, with idealized, effectively unbounded integers and unconstrained inputs. The artifact that runs on an open-hardware PLC is different: it executes on a resource-constrained Microcontroller Unit (MCU) where a machine word is only 16 bits on the 8-bit AVR boards (Arduino Uno/Nano/Mega), and physical inputs arrive through an Analog-to-Digital Converter (ADC) of finite resolution (a 10-bit ADC yields integer readings in [0 , 1,023] ). Arithmetic that is exact in the abstract model can therefore overflow on the device, and a property proved on the logic model need not hold on the deployed binary. We call this discrepancy the deployment gap .The gap cuts both ways, and both directions matter for assurance. A program can be unsafe on the device yet pass abstract verification (a missed defect); and – less obviously but, as we show, far more common – naively accounting for the word width without modelling the hardware inputs makes verification unsound in practice : the verifier explores sensor values no ADC can produce (e.g. IN = −32 ,764 ), reports overflows that can never occur, and buries any real finding under false alarms. On 123 real third-party programs, we measured a 44 % false-alarm rate (54/123) with zero genuine defects found this way (§8). 

1.2 Contributions 

This paper closes the gap with a method and its open-hardware instantiation. Our thesis is “what you verify must be what runs on the device” , which requires modeling both the target word width and the hardware-realizable input ranges. 1. Hardware-faithful verification of IEC 61131-3 (§5). A method that, given a program P and a declarative Hardware Abstraction Layer (HAL) descriptor H (word width, ADC/Pulse-Width Modulation (PWM) resolution, I/O binding), constructs a sound deployment model Mdev (P, H ): arithmetic is interpreted at the target word width and every input is constrained to its hardware-realizable range. We give the construction, a soundness argument, and a gap-diagnosis procedure that flags hardware-induced defects with physically realizable witnesses. Methodologically, this transfers to the PLC domain the target-model discipline long established for safety-critical embedded C (Frama-C’s machine model [ 12 ], Astrée [ 13 ]), with the novelty that the HAL model is derived 

automatically from the program’s PLCopen I/O addresses. 2. ESBMC-Arduino: an Arduino instantiation (§6). A HAL library whose parameters are sourced from the official Arduino cores [ 14 ], and a realization of the input-range model as a lightweight ( ≈40-line) hook in the Efficient SMT-based Context-Bounded Model Checker (ESBMC) LD frontend – the Satisfiability Modulo Theories (SMT) backend and k-induction engine are reused unchanged. 3. Evaluation on real and controlled corpora (§8). On 123 real third-party IEC 61131-3 programs, naive 16-bit overflow checking yields a 44 % false-alarm rate and zero true findings; the HAL annotator eliminates all 54 false alarms (54 →0) while preserving every robustness proof. A controlled corpus demonstrates the genuine width-dependent defects the method detects (with realizable witnesses) and precisely characterizes which arithmetic patterns are vulnerable, thereby establishing, honestly, that the gap is real but rare in the public corpora we could ingest. 

1.3 Scope 

We target the integer/Boolean fragment of IEC 61131-3, deployed via open-hardware toolchains, and evaluate at the model level (no hardware-in-the-loop). Non-linear floating-point function blocks (e.g. Proportional-Integral-Derivative (PID), transcendental sensor linearization) are outside the deployment-gap claim and are soundly over-approximated (§5); we discuss this and other limitations in §10. 2ESBMC-Arduino: Closing the Deployment Gap for Open-Hardware PLCs A P REPRINT 

Why it Matters 

Open-hardware PLCs are not toys: they run real water, lighting, and process-control logic, and they are the default substrate for the academic ICS security testbeds on which much of the field’s empirical work depends. A safety or security argument built on abstract verification of such a system is only as sound as the assumption that the abstract model matches the device – an assumption this paper shows to be false in a precise, reproducible way, and then repairs. The repair is cheap (a four-field descriptor and a 40-line hook) and reuses a mature verifier, so it is realistic to adopt rather than a clean-slate proposal.

# 2 Background 

2.1 The PLC Scan Cycle 

A PLC executes a deterministic scan cycle : it reads all inputs into a process image, executes the user program once from first rung to last, writes the output image to the actuators, and repeats [ 9]. The ESBMC-PLC line encodes this as a  

> while(true)

loop in which inputs are re-sampled each iteration (an open-world sensor model) nondeterministically, output coils are persistent state, and safety properties are injected as assertions; k-induction then proves invariants for all future scans. We reuse this encoding and add a faithful model of the hardware boundary that the scan cycle crosses on a real device. 

2.2 Open-Hardware PLCs and the OpenPLC Toolchain 

Open-hardware PLCs run IEC 61131-3 programs on commodity microcontroller boards. OpenPLC [ 1, 2 ] is the domi-nant open-source example: its editor exports PLCopen XML, and its runtime compiles the program (via MATIEC [ 15 ]) to C, which is then cross-compiled for Arduino-, ESP32-, or Raspberry Pi-class targets. Arduino OPTA [3] is Arduino’s industrial micro-PLC, programmed in IEC 61131-3 via the Arduino PLC IDE [ 4]; CONTROLLINO [5 ] and 

Industrial Shields M-Duino [6 ] are Arduino-compatible industrial PLCs commonly driven by OpenPLC. All reduce a vendor-neutral program to C plus a hardware binding (the object we verify). 

2.3 The Hardware Abstraction Layer (HAL): Word Width, Pin Binding, ADC, and PWM 

The HAL connects logical IEC 61131-3 variables to physical peripherals. Each Boolean %IX /%QX address binds to a General-Purpose Input/Output (GPIO) pin; analog inputs are read through an ADC of finite resolution (typically 10–14 bits) and scaled; analog outputs are produced by PWM. Two facts of this layer drive the deployment gap: (i) the MCU’s native word width – 16-bit int on AVR-class boards – so computations exact in the abstract model can overflow; and (ii) the bounded input domains – an n-bit ADC reading lies in [0 , 2n − 1] , a digital input in {0, 1} – which the abstract model omits entirely (Fig. 1). 

2.4 PLCopen XML 

LD programs are exchanged as PLCopen XML in textual and graphical encodings, reconstructed into rungs by the ESBMC-GraphPLC Depth-First Search (DFS) resolver [ 8]. Open-hardware editors export the graphical encoding; crucially, the variable declarations carry explicit %IX /%IW addresses, which we exploit as the ground truth for HAL input binding. The IEC 61131-3 direct-addressing scheme is itself the bridge between logic and hardware: a location prefix names the memory area ( I input, Q output, M internal) and a size prefix names the width ( X bit, B byte, W word, D double word). Thus %IX0.0 is a single digital input bit, while %IW0 is a 16-bit analog input word; on an open-hardware target, the former is wired to a GPIO pin read by digitalRead and the latter to an ADC channel read by analogRead . 2.6 Machine Models of Arduino-Class Targets 

The single most consequential difference between the abstract model and the deployed artifact is the machine word width , fixed by the target ABI. On the 8-bit AVR architecture that powers the Arduino Uno, Nano, Mega, and the Arduino-compatible CONTROLLINO Micro/Maxi and Industrial Shields M-Duino, the C int is 16 bits (range 

[−32 ,768 , 32 ,767] ), following the avr-gcc ABI. On the 32-bit ARM Cortex-M cores that power the Arduino Due, Zero, MKR, UNO R4, and the industrial OPTA, int is 32 bits . Table 2 (§6) summarizes the relevant parameters. The consequence is stark and routinely surprising to engineers: the multiplication x*100 that is harmless on a Due overflows on a Uno once x exceeds 327 , because the result leaves the 16-bit range and wraps under two’s-complement semantics. Analog peripherals compound this: the AVR ADC is 10-bit (readings 0 . . . 1,023 ) and analogWrite drives an 8-bit PWM duty ( 0 . . . 255 ), whereas the UNO R4 offers a 14-bit ADC; the combination of a narrow word and a wide sensor range is precisely the regime in which scaling arithmetic overflows. 2.7 Bounded Model Checking and k-Induction for PLCs 

Our backend is ESBMC [ 16 , 17 ], an SMT-based Bounded Model Checking (BMC) engine. BMC unrolls the scan loop a bounded number of times k and encodes “does a property violation occur within k scans?” as an SMT formula whose satisfying assignment, if any, is a concrete counterexample trace [ 17 ]. Because ESBMC models machine integers as fixed-width bit-vectors , arithmetic overflow and two’s-complement wrap-around are reproduced bit-precisely – the property of the solver we rely on to detect width-dependent defects, and the reason a faster word-agnostic abstraction would be unsound here. Plain BMC is a falsifier: it finds bugs up to depth k but does not prove their absence. To obtain unbounded proofs over the non-terminating scan loop, ESBMC uses k-induction [18 , 17 ]: a base case checks the first k scans, and an inductive step shows that if the property holds for k consecutive scans it holds for the next, with internal state hacked. For a per-scan, state-independent property such as “the scaling arithmetic does not overflow,” the inductive step typically closes at small k; for properties whose proof requires reasoning about loop-carried internal state – as in function blocks with internal counters – the inductive step may fail to close, yielding an inconclusive  

> unknown

verdict rather than safe . This distinction explains both why our method proves many programs robust and why a residual set remains unknown (§8). Overflow itself is encoded as a verification condition: for a subtraction a-b 

of signed type, ESBMC asserts !overflow("-", a, b) , whose negation the solver tries to satisfy within the modeled input ranges (Fig. 3).

We situate the work along three axes – verification of IEC 61131-3, target-faithful analysis of embedded software, and code-generation correctness – and summarize the positioning in Table 1. 

3.1 Verification of IEC 61131-3

A complementary and fast-growing line establishes precise formal semantics for IEC 61131-3 and analyzes richer execution settings: • K-ST gives an executable semantics for ST [21]; • Lee and Bae [22] formalize multitask ST with preemption; • Closest to our cyber-physical concern, Lee and Bae [23] formalize networked PLCs interacting with physical environments; • FRET-based monitor synthesis extends PLCverif with requirement formalization [24]; • Cooperative verification orchestrates multiple backends via CoVeriTeam [25]; • Ladder logic has been formalized for fault-tolerance analysis [26]. Industrial practice mirrors this – an SMT-based ladder-to-model-checker translation is patented for network anomaly detection [27]. The ESBMC-PLC line targets the ESBMC backend specifically: • ESBMC-PLC [7] introduced a native textual PLCopen XML LD frontend with k-induction; • ESBMC-GraphPLC [8] added the graphical encoding via a DFS rung resolver; • ESBMC-PLC+ [ 9] added ST/SCL ingestion through MATIEC and scan-cycle-accurate semantics for timer, counter, and edge function blocks. What unites all of these tools – commercial and academic alike – is that they verify an abstract model of the control logic: integers are mathematical (or fixed at a host width chosen for convenience, typically 32 bits), and inputs are unconstrained. None models the word width of the deployment target or the bounded domains of the physical inputs. The deployment gap is therefore orthogonal to and composable with each of these efforts; our method could, in principle, be layered onto PLCverif or Arcade.PLC as readily as onto ESBMC. 

3.2 Target-Faithful Analysis of Embedded Software 

The discipline we transfer to the PLC domain is well established for handwritten embedded C. • Frama-C [ 12 ] carries an explicit machine model selected with -machdep (e.g. avr fixes a 16-bit int ), and its value analysis obtains input ranges from ACSL contracts written by the Engineer. • Astrée [ 13 ], the abstract interpreter used in avionics certification, reliably reports integer and floating-point overflow in the target’s arithmetic and derives input bounds from volatile-range and known_fact annotations. • Industrial static analyzers in the same lineage detect overflow under a configured target model. All of these “deal with the gap” in exactly the manner we advocate – target machine model plus bounded inputs – which is precisely why our approach is defensible rather than speculative. Two differences distinguish our work. 1. The object of analysis is IEC 61131-3, not C: the Engineer never sees or writes the generated C, so any hardware model must be expressed at, or recovered for, the PLC level. 2. And centrally, those tools require the input ranges to be supplied manually (as contracts or annotations), whereas we derive them automatically from the IEC 61131-3 direct-addressing scheme: a %IW location is, by the standard, an analog input word, and the board descriptor fixes its ADC width. The Engineer writes nothing. 

3.3 Code-Generation Correctness 

A separate body of work establishes that a code generator preserves semantics from model to executable: • CompCert [28] for general-purpose C-to-assembly. • Qualified generators such as SCADE KCG in the model-based-design world. These guarantees are real but orthogonal: even a perfectly verified translation does not prevent a program’s own 

arithmetic from overflowing on a small target, because the overflow is a property of the source semantics under the target word width, not of the translation. # 4 The Deployment Gap, Formally 

4.1 Physical Process and Fault Model 

An open-hardware PLC is one element of a closed cyber-physical loop (Fig. 4). A physical process – a tank, a burner, a conveyor – is observed by sensors whose continuous signals are digitized by the MCU’s ADC into integer codes; the scan-cycle logic reads those codes, computes in the MCU’s finite-width integer arithmetic, and drives actuators through digital pins and PWM, which in turn act on the process. A safety property is ultimately a property of this loop – “the high-level alarm actuates before the tank overflows” – realized in software as an assertion over the program variables that bind to the physical I/O. The cyber and the physical meet at exactly two surfaces: the ADC that turns a bounded physical quantity into an integer (so each input has a hardware-realizable range , not the full machine word), and the integer arithmetic that runs at the target’s word width (so a computed actuation command can wrap). These two surfaces are the locus of the deployment gap.

4.1.1 Fault model 

We do not assume a malicious adversary; the fault we target is a latent computational fault induced by legitimate physical inputs . A program that is correct in the abstract model can, on a narrow-word target, compute the wrong actuation for a perfectly ordinary sensor reading – an integer overflow in an ADC-to-engineering-unit conversion silently inverts a comparison and suppresses a safety action. The fault is triggered by inputs the environment routinely produces (within D(a), the bounded domain D of physically realizable values for analog input a), is invisible at the PLC-logic level existing verifiers analyze, and manifests only once the logic is composed with a specific board’s HAL. Dually, an unconstrained input model fabricates faults the environment can never trigger (inputs outside D(a)): these are the phantom alarms of §8. A faithful model must therefore admit exactly the physically realizable inputs – no more, no less. 

4.1.2 Assumptions 

We assume the standard PLC scan-cycle timing model (§5.1): inputs are sampled at the top of each scan and held constant during rung evaluation, so we reason about values, not intra-scan transients, and we do not model continuous plant dynamics, sensor noise, or actuator latency – the property is the per-scan logical assertion, and the environment is the open-world set of realizable input vectors. 5.1 Scan-Cycle Execution Model 

We model a PLC program as a transition system T = ( S, s 0, R ) over a state S that partitions variables into inputs 

I, outputs O, and internal state V. One scan is the relation scan ⊆ S × S that (i) re-samples every input i ∈ I ,(ii) executes the program body once, updating O and V, and the system iterates scan indefinitely. A safety property φ is an assertion required to hold after every scan. Two parameters of this model are left implicit by existing verifiers but fixed on a real device: the interpretation of arithmetic operators (the word width w at which +, −, × are evaluated, with two’s-complement wrap outside [−2w−1, 2w−1)) and the input domains (the set D(i) ⊆ Z from which each i ∈ I 

may be sampled). The abstract model Mabs takes w large and D(i) = Z; the deployment model Mdev (P, H ) takes both from H (Fig. 7.

6.2.1 Where the Hook Sits

The annotator attaches at exactly these two sites, immediately after each nondeterministic assignment, so that the bound applies to the freshly sampled value on every scan.

8.2 RQ1: The Gap Is Real but Rare The Gap is Real (Controlled Corpus) 

The same program and property flip from safe under Mabs to unsafe under Mdev on the 16-bit boards, each with a realizable witness (Table 4); the correctly-coded _ok variants never flip, on any board, confirming that the flips track a genuine hardware-induced overflow rather than a modelling artifact. Which Patterns are Vulnerable 

The flip condition is precise and instructive for practitioners. With an input bounded to a 10-bit ADC ( v ≤ 1,023 ), a 16-bit overflow ( > 32 ,767 ) requires the arithmetic to magnify v past the word range: multiply by a constant ≳ 33 (e.g. percentage or milli-unit scaling, v × 100 ), accumulation of many readings, or squaring . Conversely, multiply-by-small-constant ( v × 2), additive offsets ( v ± c), and comparisons all remain within range and are provably safe – which is why the _ok variants, and the bulk of real code, do not flip. 

The Gap is Rare in the Corpora We Surveyed 

We surveyed the real corpus and a larger pool of public Structured-Text libraries for the vulnerable pattern. Within these corpora, the pattern is essentially absent: across 341 ST programs in a verification benchmark library, none 

contained a multiply-by-a-three-or-more-digit constant; the integer arithmetic that does occur is dominated by ×2,offsets, clamping, and comparisons – operations that bounded inputs keep safe. The public PLC code we could ingest is overwhelmingly Boolean and timer logic with small-magnitude integer arithmetic. We therefore report, honestly, that in these corpora the deployment gap is genuine and sound to detect but infrequent ; the analog-scaling regime in which it bites (e.g. ADC-to-engineering-unit conversion on a narrow-word board) is real but not the common case here. We caution that this prevalence finding is scoped to the integer/Boolean LD programs our frontend ingests (§10); analog-heavy process-control corpora may exhibit it more often. This finding reframes the contribution: the value of hardware-faithful verification lies less in bug count than in the soundness it restores (RQ2–RQ3).

A genuine limit remains, and we state it plainly. The 54 eliminated false alarms become unknown rather than safe : the spurious overflow is gone, but the prover does not prove those programs safe. The cause is k-induction incompleteness on function-block-heavy programs. Tracing one such case, ESBMC unwinds an internal function-block loop to the default bound, reports that “the forward condition is unable to prove the property,” and that the inductive step does not close because it must havoc-loop-carry internal state. This is a property of the prover, not of the input model: 37 of these programs are already unknown in the naive run, and enabling the annotator neither helps nor harms their proof (it only removes the false unsafe ). bounded-proof configurations did not change the outcome. Closing these – via loop-invariant inference or compositional 

k-induction – would raise the robustness-proof count above the current 32 and is the most valuable direction for future work (§11). The headline soundness result (54 →0 false alarms) is independent of this limit. 

# 9 Discussion 

The strongest contribution of this work is not a bug count but a soundness result: open-hardware PLC verification that accounts for the word width but ignores the hardware input domains is unusable (44% false alarms, zero true findings), and a small, automatically-derived HAL model restores usability (54 →0 false alarms) while preserving every proof. This mirrors how target-faithful embedded-C analysis is judged [ 13 ]: value lies in sound proofs and the absence of false alarms, not in headline defect counts. An analyzer that cries wolf is switched off; one that issues only realizable reports is trusted. RQ1 lets us tell practitioners precisely when to worry. The deployment gap manifests when bounded-range sensor data is magnified by integer arithmetic beyond the narrow machine word: ADC-to-engineering-unit scaling ( raw * span/range ), running sums and integrators, and squared terms, all deployed on a 16-bit AVR board. Boolean interlocks, timers, counters with small presets, and offset/threshold logic are safe by construction and need no special attention. This is actionable guidance independent of our tool: an engineer porting a recipe or PID-like computation from a 32-bit controller to a Uno-class board should widen intermediates to DINT or verify with a hardware-faithful model. The embedded-C analyzers that pioneered target-faithful analysis require the Engineer to supply input ranges as contracts or annotations. In the PLC setting, that burden would fall on an automation engineer unfamiliar with formal specification, and – for a corpus of 123 programs – would dominate the effort. Recovering the bounds from the IEC 61131-3 direct-addressing scheme removes the burden entirely: the %IW /%IX address is the specification of the input’s nature, and the board descriptor supplies the width. The Engineer writes nothing, which is what makes the method deployable at the corpus scale. Although we instantiate for Arduino, neither the method nor the implementation is Arduino-specific. The HAL descriptor is a four-field record; retargeting to another open-hardware family (e.g. an ESP32-based controller, or a 32-bit industrial soft-PLC) is a matter of supplying its word width and ADC/PWM resolutions. The same hook, applied with a 32-bit width, reproduces the abstract behavior, so the method strictly generalizes existing abstract verification rather than replacing it. 

# 10 Threats to Validity 

The naive baseline and the annotated runs are produced by the same binary, differing only by the HAL_ANNOTATE 

toggle, so the measured 54 →0 change is attributable to the input model alone and not to incidental differences in tool configuration. Verdicts agree between –16 and –32 on all 54 naive failures, an independent check that those reports are input-driven rather than width-driven. Modelling an integer IEC input as a 10-bit ADC source is the AVR HAL assumption encoded in H. An input physically wired to a wider ADC (e.g. a 14-bit UNO R4 channel), to a digital word, or to a network variable would carry a different domain; the descriptor is configurable per address, and a mis-specified bound would be unsound in the same way a wrong -machdep would be for Frama-C. In this study, every integer input is treated uniformly as a 10-bit analog source, the conservative common case for the AVR boards; finer per-address domains are a straightforward extension once richer I/O metadata is available. 91/123 programs are unknown owing to k-induction limits on loop-bearing function blocks (§8), bounding the robustness-proof count to 32. We emphasize this is a backend limitation orthogonal to the contribution: it is present with and without the input model, and the soundness result (no false alarms, no suppressed real defects) holds regardless. It does, however, mean we cannot claim “123 programs proven robust,” only “54 false alarms removed and 32 robustness proofs retained.” The deployment gap is rare in the public corpora we could ingest, which are dominated by Boolean/timer logic. Analog process-control corpora (e.g. ICS testbeds in REAL arithmetic) where high-magnitude scaling – and hence the gap – may be more common fall outside our current frontend (REAL/non-linear support is partial) and are not represented; our prevalence claim is therefore scoped to integer/Boolean LD programs. The two datasets are third-party and pinned, but they are not the universe of PLC code. The non-linear abstraction is sound but imprecise: a property that genuinely depends on a PID output cannot be proved under it (it will be unknown or spuriously unsafe on the abstracted output). Hence, the method is most precise on the 17 ESBMC-Arduino: Closing the Deployment Gap for Open-Hardware PLCs A P REPRINT 

integer fragment it targets. Finally, we validate at the model level; we do not execute the verified binaries on physical boards, so the soundness guarantee is relative to the faithfulness of H and the frontend’s semantics rather than measured against silicon. Hardware-in-the-loop confirmation is future work. 

# 11 Future Work 

Several directions follow directly from the limitations above. The most valuable next step is to convert the 91 unknown verdicts into safe robustness proofs, by strengthening 

k-induction over function-block internal state – loop-invariant inference, or compositional k-induction that verifies each function block once and composes the results, avoiding the monolithic unwinding that currently fails to close. The same framework expresses mechanisms (M1) and (M2): modeling ADC quantization precisely (representable-code sets rather than intervals) would let the method find threshold-straddling defects, and modeling pin binding from the  

> io_map

would catch aliasing faults; both are properties over the existing model rather than new machinery. A fixed-point abstraction of non-linear blocks would be more faithful than the current nondet over-approximation – and is itself how such computations are often realized on an MCU – enabling sound, precise verification of the analog programs presently out of scope, and opening the ICS-testbed corpora where the gap may be more prevalent. Per-address ADC widths and engineering-unit scaling recovered from the project file would replace the uniform 10-bit assumption with exact per-input domains. Finally, executing the verified programs on physical Uno/Due boards and confronting the synthesized witnesses (e.g. 

adc = 898 ) with measured behavior would close the loop between the model-level guarantee and the device, and is the natural validation of the soundness theorem. 

# 12 Conclusion 

We presented hardware-faithful verification of IEC 61131-3 on open hardware – a declarative HAL descriptor and a sound lowering that models the target word width and hardware-realizable input ranges – as well as its Arduino instantiation, ESBMC-Arduino. On 123 real programs, naive width-aware verification is unusable (44% false alarms, zero true findings); the automatically derived HAL input model eliminates all false alarms while preserving every proof, and a controlled corpus characterizes the genuine, if rare, width-dependent defects that the method soundly detects.

Lemma 1 (Non-linear over-approximation) . Replacing the body of a non-linear function block by a fresh nondetermin-istic value constrained to the block’s declared output range yields a transition relation whose reachable states form a

The abstracted relation, therefore, contains the original one and adds behaviors.
