## 摘要
OpenPLC、Arduino OPTA、CONTROLLINO和Industrial Shields M-Duino将IEC 61131-3引入用于实际自动化和工业控制系统（ICS）安全研究的低成本微控制器。现有的IEC 61131-3开源验证器（包括ESBMC-PLC）通过一个带有理想无界整数的抽象扫描周期模型来证明安全性。板级组件运行在资源受限的微控制器单元（MCU）上，具有16位字长（8位AVR Arduinos），并且通过有限分辨率的模数转换器（ADC）读取传感器。我们表明，这种部署差距使得朴素的宽度感知验证变得不可靠：在123个实际程序中，检查16位溢出而不考虑硬件输入模型会产生44%的误报（54/123），并且没有发现真正的缺陷，因为它探索了ADC无法生成的传感器值。由于差距存在于计算与物理过程相遇之处——一个由有限宽度算术缩放为驱动命令的有界传感器读数——溢出可能会悄无声息地抑制安全动作，例如高级报警。无界输入模型会伪造环境无法触发的报警。我们提出了针对开放硬件上IEC 61131-3的硬件忠实验证：一个声明式硬件抽象层（HAL）描述符（宽度、ADC/PWM分辨率、I/O绑定）以及一种合理的降级方法，该方法在目标宽度上解释算术运算，并将输入约束到硬件可实现的范围。我们将其实例化为Arduino上的ArduinoTool，从官方内核中推导出HAL参数，并在ESBMC梯形图（LD）前端中实现输入范围模型。在123个程序语料库上，HAL注释器消除了所有54个误报，同时保留了鲁棒性证明，而受控语料库则展示了它通过可实现的证据检测到的罕见的宽度相关缺陷。

## Abstract
OpenPLC, Arduino OPTA, CONTROLLINO, and Industrial Shields M-Duino bring IEC 61131-3 to low-cost microcontrollers used in real automation and industrial control system (ICS) security research. Existing open-source verifiers for IEC 61131-3, including ESBMC-PLC, prove safety over an abstract scan-cycle model with idealized unbounded integers. The board artifact runs on a resource-constrained microcontroller unit (MCU) with 16-bit words (8-bit AVR Arduinos), and sensors are read via a finite-resolution analog-to-digital converter (ADC). We show this deployment gap makes naive width-aware verification unsound: across 123 real programs, checking 16-bit overflow without a hardware input model yields 44% false alarms (54/123) and finds no genuine defects, because it explores sensor values no ADC can produce. Since the gap lies where computation meets the physical process - a bounded sensor reading scaled by finite-width arithmetic into an actuation command - an overflow can silently suppress a safety action, such as a high-level alarm. An unbounded input model fabricates alarms that no environment can trigger. We present hardware-faithful verification for IEC 61131-3 on open hardware: a declarative hardware abstraction layer (HAL) descriptor (width, ADC/PWM resolution, I/O binding) and a sound lowering that interprets arithmetic at target width and constrains inputs to hardware-realizable ranges. We instantiate it for Arduino as ArduinoTool, deriving HAL parameters from official cores and realizing the input-range model in the ESBMC Ladder Diagram (LD) frontend. On the 123-program corpus, the HAL annotator eliminates all 54 false alarms while preserving robustness proofs, and a controlled corpus demonstrates the rare width-dependent defects it detects with realizable witnesses.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有用于 IEC 61131-3 的开源形式验证工具（如 ESBMC-PLC、PLCverif）均基于**抽象扫描周期模型**，假设理想化的无界整数和不受约束的传感器输入。然而，实际部署在开放硬件 PLC（如 Arduino Uno/Nano/Mega，使用 8 位 AVR MCU）上的程序运行在 **16 位字长**的微控制器上，且传感器读数通过**有限分辨率 ADC**（例如 10 位 ADC，值域 [0,1023]）输入。这种“抽象模型 vs 真实硬件”之间的差异被称为**部署鸿沟**（deployment gap）。

- **研究动机**：朴素地仅考虑字宽（16 位）而不建模输入范围会导致两种害处：  
  - **伪缺陷**（phantom alarms）：验证器会探索 ADC 无法产生的输入值（如 IN = -32764），产生大量假警报。  
  - **漏报缺陷**（missed defects）：程序在抽象模型下安全，但在实际硬件上因整数溢出而失效（例如 ADC 读数经百分比缩放后溢出，导致报警被抑制）。  
  迫切需要一种既能捕捉真实硬件缺陷、又不会触发大量假警报的形式验证方法。

### 2. 论文提出的方法论

- **核心思想**：**“你所验证的必须与设备上运行的完全一致”**，即对目标硬件的字宽和输入范围进行精确建模。  
- **关键技术细节**：  
  1. **声明式硬件抽象层（HAL）描述符**：一个四元组 ⟨w, b_adc, b_pwm, μ⟩，其中 w 是机器字宽，b_adc 和 b_pwm 分别是 ADC 和 PWM 的位分辨率，μ 是 I/O 绑定。  
     - 输入设备根据 IEC 61131‑3 的直接地址编码区分：`%IX0.0`（数字输入，域 {0,1}）、`%IW0`（模拟输入，域 [0, 2^b_adc -1]）等。  
  2. **忠实降低（Faithful Lowering）**：在 ESBMC 的梯形图（LD）前端中，对每个输入变量重新采样后立即插入 `assume(0 ≤ v ≤ ADCMAX)`，将算术运算解释为目标字宽（如 16 位）。  
  3. **非线性抽象**：对于 PID、温度传感器线性化等浮点/超越函数块，用约束在其声明输出范围内的非确定性值替换体部（保守近似）。  
  4. **验证过程**：复用 ESBMC 的 SMT 后端的位精确 BMC（有界模型检查）和 k‑归纳引擎，生成的部署模型 Mdev(P,H) 得到物理可实现的违反证据。

- **算法流程（文字说明）**：  
  1. 设定机器字宽为 HAL.w → 所有整数运算按 w 位比特向量处理（精确匹配硬件）。  
  2. 对每个输入变量 i，在其重新采样后添加 `assume(i ∈ [0, 2^H.b_adc -1])`。  
  3. 对每个模拟输出，将其值钳位到 [0, 2^H.b_pwm -1]。  
  4. 对每个非线性函数块，用非确定性值替换其体部。  
  5. 运行 ESBMC 执行溢出检查和 k‑归纳证明。

### 6. 论文的主要结论与发现

- **朴素宽度感知验证不可用**：在 123 个真实程序中，仅设置 16 位溢出检查导致 54 个（44%）伪 unsafe 裁决，未发现任何真正缺陷。  
- **HAL输入模型恢复声纳**：开启后所有 54 个误报被消除（54→0），全部 32 个先前的安全证明被保留；但 91 个程序变为 unknown（k‑归纳不闭合）。  
- **真实鸿沟存在但罕见**：受控语料库验证了当 ADC 读取值范围较大、且算术放大幅度足够（如乘 100）时，16 位溢出可导致安全动作被静默抑制。然而在可获得的公共语料中此类模式很少见。  
- **方法迁移性**：将嵌入式 C 领域的目标忠实分析（如 Frama‑C、Astrée）的原则首次应用于 PLC 领域，并通过 IEC 61131‑3 直接地址方案**自动获得输入范围**（无需工程师手动注释）。

### 7. 优点

1. **最小的侵入性**：仅需在 ESBMC LD 前端增加约 40 行的输入范围注释钩子，完整复用后端 BMC + k‑归纳引擎，无需修改 SMT 求解或验证流程。  
2. **自动推导输入范围**：利用 IEC 61131‑3 的地址编码（`%IX` 数字输入，`%IW` 模拟输入）和板级描述符，工程师无需手写注释——对比 C 级工具（Frama‑C、Astrée）必须手动提供输入范围。  
3. **跨工具一致性验证**：在 CBMC 和 Frama‑C 上复现了同样模式，证明部署鸿沟不是工具特有，排除了单点失败。  
4. **实用性强**：运行时开销可忽略（<1s/程序），误报率降至零，使得工程师可以信任验证结果。  
5. **清晰的物理故障模型**：将部署鸿沟明确定义为有界传感器输入与有限字宽算术的组合，而非模糊的“代码生成错误”。

### 8. 不足与局限

1. **k‑归纳不完备**：约 74%（91/123）的程序变为 `unknown`，主要原因是函数块内部的循环体导致 k‑归纳的归纳步骤无法闭合。虽然不影响误报消除，但未能提供鲁棒性证明，这是未来需要增强的方面（如循环不变量推断、组合式 k‑归纳）。  
2. **范围仅限整数/布尔子集**：方法针对 IEC 61131‑3 的整数/布尔子集。PID、非线性传感器线性化等浮点/超越函数块被保守近似为非确定性值，可能导致对这些块的特性无法验证。  
3. **输入域假设统一**：实验中所有整数输入均保守视为 10‑bit ADC 源（AVR），若实际连接到更宽 ADC（14-bit）或数字字输入，描述符需配置更细粒度；当前研究未实验更细致的每地址分辨率映射。  
4. **缺少硬件在环验证**：结论基于模型级验证，没有在实际物理板上运行验证后的二进制，无法直接测量模型与实际设备间的偏差。  
5. **公共语料代表性有限**：主要来自水处理和克隆检测数据集，以布尔/定时器为主；模拟量密集型（如连续过程控制）程序很少，因此“鸿沟罕见”这一结论可能有偏差。  
6. **未开源完整的跨板支持**：尽管方法通用，但实现目前仅实例化于 Arduino 核心，未覆盖其他开放硬件家族（如 ESP32‑based 或工业软 PLC）的详细验证。
