## 摘要
形式化契约对于软件测试和验证至关重要，但编写它们仍然劳动密集且容易出错。LLM 为自动形式化提供了一条有希望的路径：从自然语言规范中合成可执行的断言，从而弥合非正式开发人员意图与正式可执行规范之间的差距。我们提出了 Monty：一个用于断言的自动形式化框架，它解决了断言有效性的期望和自然语言中的歧义性挑战。我们的技术基于使用一种新颖的一致性得分指标和对代码进行测试获得的断言有效性分数来过滤形式化。我们在从 22 个类似集合的 Java 类派生的 541 个断言生成任务上评估了我们的方法，并表明与简单使用 LLM 翻译断言相比，我们的技术更可靠地生成真实情况（平均精度提高多达 20 个点）。

## Abstract
Formal contracts are essential for software testing and verification, yet writing them remains labor-intensive and error-prone. LLMs offer a promising path toward autoformalization: synthesizing executable assertions from natural-language specifications and thereby bridging the gap between informal developer intent and formal executable specifications. We present Monty: an autoformalization framework for assertions that tackles the challenges of expectations of validity of assertions and ambiguity in natural-language. Our techniques are based on filtering formalizations using a novel conformance score metric and validity scores obtained from testing the code against formalized assertions. We evaluate our approach on 541 assertion-generation tasks derived from 22 collection-like Java classes, and show that our technique produces the ground truth more reliably (improving upto 20 points in precision on average) than when using LLMs naively to translate assertions.

  - (a) 如何有效评估形式化是否忠实于原意？

- **核心贡献**：提出 Monty 框架，结合 LLM、程序测试和主动学习，实现高精度的自动形式化，并公开多个基准数据集。

## 2. 方法论

- **核心思想**：不依赖单一 LLM 输出，而是生成多个候选，然后通过两个互补信号进行筛选，最后通过主动学习消除剩余歧义。
- **技术流程**：
  1. **候选生成**：使用 LLM（如 GPT-oss）在给定代码骨架和 NL 断言后，生成 5 个候选形式化断言（JML 格式）。
  2. **有效性评分**：对每个候选进行三级检查：
     - 语法检查：编译是否通过。
     - 模糊安全（Fuzz-Safety）：通过随机测试（Randoop）检查评估断言是否引发运行时异常（如空指针）。
     - 模糊语义（Fuzz-Semantic）：执行方法后检查断言是否为真（有效）或假（无效）。
  3. **一致性评分**（核心创新——子句覆盖）：
     - 先将形式化断言反向翻译回自然语言（roundtrip）。
     - 使用 LLM-as-a-Judge 比较原 NL 描述和反向翻译的描述，要求 LLM 按子句覆盖（clausal coverage）度量：
       - 完备性：反向翻译是否覆盖原 NL 所有子句。
       - 声音性：反向翻译是否引入原 NL 没有的信息。
     - 评分范围 [-1, 1]，重复 8 次取平均，阈值设为 0.6。
  4. **决策矩阵**：从所有候选中选择两个代表：
     - 有效性得分最高的候选（若通过测试）。
     - 无效性得分最高的候选（若未通过测试）。
     - 仅当一致性分数 ≥ 0.6 才入选。
  5. **主动学习**：如果存在两个候选（一个有效一个无效），生成一个区分性输入（如 null 元素），询问用户该输入是否符合预期断言，从而确定正确形式化。

- **关键技术细节**：
  - 反向翻译 prompt 要求忠实保留每个前提、界限、旧状态引用等。
  - 一致性比较 prompt 提供上下文（方法签名、文档），并指示忽略措辞差别，重点比较逻辑骨架。
  - 测试器使用 Randoop 构建对象状态，支持 `\old`、`\forall` 等 JML 操作符。

## 3. 实验设计

- **数据集**：

  - **Buggy Code**（20 对）：修改目标方法代码，使断言无效。
  - **Buggy Assertion**（66 对）：修改断言本身（误导性或错误）。
  - **Manual NL Specs**（39 对）：由一位作者手工撰写自然语言规范，对应三个新 Java 类，不展示给其他作者。

## 6. 主要结论与发现

- **Monty 显著提升可靠性**：在 C2S-Aug 数据集上，Qwen2.5-Coder 的精度从 75% 提升至 91.6%，GPT-oss 从 85.3% 提升至 89.7%（Monty-Acc），且精度提升不损失召回（召回 >93%）。
- **在小模型上效果更明显**：精度提升可达 20 个点（如 Qwen2.5-Coder 从 64% 到 85.2% 在手写 NL 集上）。
- **一致性检查是关键**：提出的子句覆盖方法 (Ours) 在 AUC 上优于所有消融变体（AUC=0.659）。
- **主动学习在少数歧义情况有效**：虽然案例少，但准确率 100%（GPT-oss 上）。说明在必要时刻减少用户负担。
- **LLM 有偏见**：对 Buggy Assertion 任务，LLM 偏向生成有效断言，导致召回低（Qwen2.5-Coder 仅 26.3%），说明工具仍需改进。

## 7. 优点

- **方法创新性**：提出“子句覆盖”一致性评分，结合 roundtrip 翻译和细粒度子句比对，更好捕获语义等价性。
- **处理歧义策略**：明确区分有效和无效形式化，并使用主动学习消除歧义，不预设断言有效性。
- **实验设计全面**：涵盖合成/手动、有效/无效、多种模型、多种消融，且公开数据集。
- **实际可用性**：精度超过 90%（对大模型），达到实用门槛；代码开源（暗示）。
