## 摘要
AI加速器算子被编译成多级流水线程序，其中DMA、向量、矩阵和标量单元在共享片上缓冲区上并发执行。缺失或错位的同步原语会导致硬件可见的数据竞争，而仿真和黄金测试都无法发现这些竞争，因为两者都没有建模加速器的跨单元可见性语义。我们将加速器流水线程序形式化为一种受限的并发语言，定义了一个参数化的硬件事件语义，包含三种排序关系——程序顺序、同步顺序和屏障顺序——并将正确性问题归结为屏障充分性：即同一缓冲区上的每个跨单元写-读对是否由happens-before关系排序。这里的“屏障”表示模型中的一种抽象排序原语，涵盖了供应商流水线屏障、硬事件同步以及等效的前端归一化同步点。我们证明了屏障充分性在$O(|E|^2)$时间内可判定，并且我们的检查器在建模语义下既是正确的也是完备的。我们实现了AccelSync，一个静态验证工具，通过仅更改硬件模型实例化到Ascend 910B2和Cambricon MLU370上。在来自CANN算子库的6292个生产内核上，AccelSync识别出3个先前未知的同步风险——其中一个匹配我们在特定工具包/驱动配置（CANN 8.0.RC3）下在Ascend 910B2上观察到非确定性输出的风险类别，尽管在后续驱动升级后该观察结果无法复现——在120个LLM生成的内核上，它标记出19.2%的缺陷率（95%置信区间：[13.0%，27.4%]）。对688个非等价变异体的变异研究实现了100%的检测率，而正面比较显示AccelSync能够检测到华为运行时清理工具msSanitizer遗漏的风险，且每个内核的成本降低了400倍。

## Abstract
AI accelerator operators are compiled into multi-stage pipeline programs where DMA, vector, matrix, and scalar units execute concurrently on shared on-chip buffers. A missing or misplaced synchronization primitive introduces hardware-visible data races that escape both simulation and golden testing, because neither models the accelerator's cross-unit visibility semantics. We formalize accelerator pipeline programs as a restricted concurrent language, define a parameterized hardware event semantics with three ordering relations -- program order, synchronization order, and barrier order -- and reduce the correctness question to barrier sufficiency: whether every cross-unit write-read pair on the same buffer is ordered by happens-before. Here "barrier" denotes an abstract ordering primitive in the model, covering vendor pipe barriers, hard-event synchronization, and equivalent frontend-normalized synchronization points. We prove that barrier sufficiency is decidable in $O(|E|^2)$ time and that our checker is both sound and complete under the modeled semantics. We implement AccelSync, a static verification tool instantiated for Ascend 910B2 and Cambricon MLU370 by changing only the hardware model. On 6,292 production kernels from the CANN operator library, AccelSync identifies 3 previously unknown synchronization hazards -- one matching a hazard class for which we observed nondeterministic outputs on Ascend 910B2 under a specific toolkit/driver configuration (CANN 8.0.RC3), though this observation was not reproducible after a subsequent driver upgrade -- and on 120 LLM-generated kernels it flags a 19.2% defect rate (95% CI: [13.0%, 27.4%]). A mutation study on 688 non-equivalent mutants yields 100% detection, and a head-to-head comparison shows AccelSync detects hazards that Huawei's runtime sanitizer msSanitizer misses, at 400x lower cost per kernel.

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- 将加速器流水线程序形式化为一种**受限并发语言**，通过事件语义和三种排序关系（程序序po、同步序so、屏障序bo）构建happens-before图，将正确性归约为**屏障充分性（barrier sufficiency）**：每个跨单元或跨阶段写-读对必须被happens-before排序。
- “屏障”是抽象排序原语，可映射到不同硬件的具体同步原语（如PipeBarrier、SetFlag/WaitFlag、__sync()等）。

### 关键技术细节
- **加速器流水线程序定义**：程序由多个顺序阶段组成，每个阶段含写、读、入队、出队、屏障操作。阶段间通过FIFO队列通信，阶段内多个执行单元并发。
- **硬件事件语义**：
  - 事件集E = { (kind, buffer, stage, timestamp, unit) }
  - 程序序(po)：同一阶段同一单元内按时间顺序排序；队列事件具有阶段全局排序。
  - 同步序(so)：FIFO匹配入队-出队对，建立跨阶段可见性。
  - 屏障序(bo)：由硬件模型定义的覆盖规则，例如“VPU写→标量读”需存在对应屏障事件在写之后、读之前。
  - happens-before = (po ∪ so ∪ bo)⁺。
- **屏障充分性定义**：对所有跨单元或跨阶段的写-读对，若写→hb读，则安全；否则为风险。
- **判定算法（Algorithm 1）**：提取事件 → 构建po、so、bo边 → BFS计算传递闭包（O(|E|²)） → 检查所有跨单元写-读对是否被hb覆盖。输出所有未覆盖对。
- **证明**：屏障充分性在O(|E|²)时间内可判定，且在建模语义下检查器是正确且完备的。

## 7. 优点

- **理论创新**：首次形式化加速器流水线程序的同步覆盖问题，并证明可判定性与多项式复杂性。
- **轻量高效**：基于happens-before图可达性，而非状态空间探索，单核CPU即可秒级审计数千内核。
- **硬件无关性**：硬件模型参数化，换平台只需修改模型参数和前端。
- **跨层契约**：连接编译器生成的流水线结构与硬件可见性规则，弥补了传统验证工具的空白。
- **实用性**：支持多种部署模式（源码、IR级），可集成到CI流程；提供可操作诊断（指出哪个缓冲区、哪个单元对缺失同步）。

## 8. 不足与局限

- **完备性范围**：完备性仅相对于定义的硬件事件语义，而非物理硬件。硬件可能存在未文档化的排序保障，导致假阳性；反之，微架构效应可能提供额外顺序，使模型保守。
- **模型范围**：不处理动态控制流（数据依赖的流水线深度）或运行时计算的同步标识符。
- **前端依赖性**：支持新硬件平台需要前端工程工作来降级其语法和同步习惯用法。
- **假阳性实例**：MLU370前端初始报告了10个假阳性（由于未建模组合屏障、别名/偏移分析缺失、无法跨循环边界追踪同步），说明前端成熟度有待提高。
- **硬件验证不完全**：VPU→标量风险类虽然在CANN 8.0.RC3上观察到非确定性输出，但驱动升级后无法复现，硬件级别的风险确认仍待进一步实验。
- **数据集局限**：LLM生成内核规模有限（120个），且变异算子仅覆盖四种模式，可能未覆盖所有同步失败类型。
