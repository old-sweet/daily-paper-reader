## 摘要
验证是集成电路（IC）开发中的主要瓶颈，消耗了近70%的工作量。虽然通用验证方法论（UVM）通过结构化验证环境提高了复用性，但构建子系统级UVM测试台并生成高质量激励仍需要大量手动编码、重复的EDA工具运行以及深厚的协议和微架构专业知识。我们提出了UVMarvel，这是一个利用大语言模型（LLM）为子系统级RTL构建UVM测试台的自动化验证框架。UVMarvel引入了中间表示（IR）和总线协议库，将异构规范转化为协议正确的子系统级UVM测试台，并采用信号追踪器和Verilog补丁库来指导基于LLM的激励优化。UVMarvel是首个能够自动构建跨主流总线协议的子系统级UVM测试台的框架，其平均代码覆盖率达到95.65%，同时将验证时间从数个人工工作日缩短至4.5小时的自动化执行。

## Abstract
Verification presents a major bottleneck in Integrated Circuit (IC) development, consuming nearly 70% of total effort. While the Universal Verification Methodology (UVM) improves reuse through structured verification environments, constructing subsystem-level UVM testbenches and generating high-quality stimuli still require extensive manual coding, repeated EDA tool runs, and deep protocol and micro-architectural expertise. We present UVMarvel, an automated verification framework that leverages Large Language Models (LLMs) to build UVM testbenches for subsystem-level RTL.UVMarvel introduces an Intermediate Representation (IR) and a Bus Protocol Library to translate heterogeneous specifications into protocol-correct subsystem-level UVM testbenches, and employs a Signal Tracker and a Verilog Patching Library to guide LLM-based stimuli refinement. UVMarvel is the first framework capable of automatically constructing subsystem-level UVM testbenches across mainstream bus protocols, and it achieves an average code coverage of 95.65%, while reducing verification time from several human working days to a 4.5-hour automated execution.

- **整体含义**：现有LLM辅助验证方法（如MEIC、UVM²）仅局限于IP级，无法处理子系统级的多接口耦合、长依赖链和复杂协议。UVMarvel是首个能够自动构建并优化子系统级UVM验证框架，覆盖APB、AHB、AXI、P-Channel、Q-Channel等主流总线协议，旨在将验证时间从数人天缩短至小时级，并达到工业级覆盖率。

## 2. 方法论

### 核心思想
- 将LLM与领域专用结构化知识（中间表示、总线协议库、信号追踪器、Verilog补丁库）相结合，分两阶段自动完成子系统级UVM验证流程。

### 关键技术细节
- **阶段a：测试平台构建**
  1. **中间表示（IR）**：从多模态规格（文本、时序图、框图）中提取五部分结构化信息：Module Name、Interface Description、Register Configuration、Timing Characteristics、Functional Description。IR为LLM提供验证所需的统一视图。
  2. **总线协议库（Bus Protocol Library）**：为每种协议提供人类可读的UVM骨架（interface、driver、monitor、agent）。LLM仅被允许修改骨架中的配置参数（如信号名、位宽、地址范围），协议控制流和握手结构保持固定，从而保证协议正确性。
  3. **生成流程**：IR选择协议 → 从库中取出对应骨架 → LLM根据IR细节定制骨架，生成完整UVM测试平台。

- **阶段b：激励优化**
  1. **覆盖率分析器（Coverage Analyser）**：将HTML覆盖报告转为紧凑文本，仅保留未覆盖项及上下文，供LLM处理。
  2. **信号追踪器（Signal Tracker）**：从种子信号（未覆盖点）出发，递归跨文件追踪依赖链，直至顶层I/O端口。算法1（单文件追踪）收集相关语句；算法2（跨文件递归扩展）合并子模块的相关语句并追踪跨模块接口。最终输出紧凑的全局依赖集G。
  3. **Verilog补丁库与补丁器（Verilog Patcher）**：将信号追踪器返回的分散语句通过模板（module shell、always block、case block等）恢复为语法完整的代码片段，即“过滤后的DUT”（Filtered DUT）。
  4. **LLM引导序列生成**：基于过滤后的DUT和未覆盖点，三种LLM（GPT-4.1、Claude 4.5、Gemini 2.5 Pro）生成新序列或waive候选点。新序列在原始DUT上仿真，错误反馈给LLM修复。
