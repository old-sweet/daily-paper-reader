## 摘要
分布式协议的正确性验证是出了名的困难。证明安全性通常需要归纳不变量，这些不变量既要蕴含期望的性质，又要被每个协议转换所保持；然而，推断这样的不变量仍然是一个主要瓶颈：现有的方法要么将协议模型限制为一阶逻辑的可判定片段，要么需要专家精心设计的模板。我们提出了IC3Syn，一个神经符号框架，它通过执行基于TLA+状态的IC3风格过程，并借助大型语言模型（LLMs）来综合归纳不变量。大体上，IC3Syn结合了一个符号IC3控制器（它将不变量综合分解为聚焦的阻塞任务）和一个LLM（它提供IC3本身在TLA+规范中所缺乏的协议级推理）。这种整合使得在不施加逻辑限制或需要手动模板的情况下，能够进行有纪律且灵活的不变量搜索。我们在29个分布式协议（涵盖共识、重配置和客户端-服务器系统）上评估了IC3Syn，并将其与Endive、IC3PO、SWISS和DistAI进行了比较。IC3Syn为所有29个协议发现了候选不变量，包括MongoLoglessDynamicRaft（MLDR），一个基于Raft的工业规模重配置协议，所有对比工具均未报告解决方案，以及一个复杂的Paxos变体。在每种情况下，在有限实例上综合的不变量在TLAPS中被证明对于完整的无界协议是归纳的，从而确立了安全性。

## Abstract
Distributed protocols are notoriously difficult to verify correctly. Proving safety typically requires inductive invariants that both imply the desired property and are preserved by every protocol transition; yet inferring such invariants remains a major bottleneck: existing approaches either restrict the protocol models to a decidable fragment of first-order logic or demand expert-crafted templates.   We present IC3Syn, a neuro-symbolic framework that synthesizes inductive invariants by executing an IC3-style process over TLA+ states with the assistance of Large Language Models (LLMs). At large, IC3Syn combines a symbolic IC3 controller, which decomposes invariant synthesis into focused blocking tasks and an LLM which provides protocol-level reasoning that IC3 alone lacks for TLA+ specifications. This integration enables a disciplined yet flexible search for invariants without imposing logical restrictions or requiring manual templates.   We evaluate IC3Syn on 29 distributed protocols spanning consensus, reconfiguration and client-server systems, and compare it against Endive, IC3PO, SWISS and DistAI. IC3Syn discovers candidate invariants for all 29 protocols, including MongoLoglessDynamicRaft (MLDR), an industrial-scale Raft-based reconfiguration protocol for which none of the compared tools reports a solution, as well as one complex Paxos variant. In each case, the invariants synthesized on finite instances are shown in TLAPS to be inductive for the full unbounded protocol, thereby establishing safety.

## 1. 核心问题与整体含义（研究动机和背景）

分布式协议的安全验证极度困难，通常需要**归纳不变量**——一个既包含初始状态、又在每个协议转换下保持、且蕴含目标安全性质的状态谓词。然而，自动推断归纳不变量是形式验证的主要瓶颈：现有方法要么将协议模型限制为一阶逻辑的可判定片段（如EPR）从而排除常见的TLA+建模模式，要么要求用户手动设计谓词模板（如Endive），将大量负担转嫁给专家。本文提出一种神经符号框架**IC3Syn**，通过将IC3（属性导向可达性）算法与大型语言模型（LLM）相结合，在TLA+规范上自动合成归纳不变量，无需逻辑限制或人工模板，使不变量发现从数周专家劳动降至自动化。

## 2. 方法论：核心思想与关键技术细节

### 核心思想
- **IC3分解**：IC3通过维护一系列状态帧（frames）逐步构造归纳不变量，将全局合成任务分解为多个**聚焦的阻塞任务**——每次针对一个具体的“坏状态”生成阻塞子句。
- **LLM补充**：LLM为TLA+规范提供协议级推理能力，为每个坏状态生成高层次的量化阻塞子句（如“所有节点已投票”），替代传统SAT推导的布尔立方体无法表达的量词约束。
- **验证后端**：Apalache（符号模型检查器）处理流程中的前沿查询和前驱查询；TLC（显式状态检查器）批量筛选候选子句，并按照覆盖坏状态数量排序。

### 关键技术细节
1. **IC3控制器**：维护帧序列 F = [{Init}, {Safety}]，每轮迭代执行：
   - **前置查询**：在当前最深层帧中寻找一个能一步到达安全违例的坏状态。
   - **前驱查询**：对坏状态进行反向搜索，若找到前驱则深入更浅帧，直到发现无前驱（即IC3的“无前驱”状态）或到达帧0（报告不安全）。
   - **阻塞子句查询**：若坏状态无前驱，先查询已有子句池；若无匹配，则调用LLM生成候选子句，经过TLC筛选和Apalache准入检查（Init ⇒ c 和 F_{k-1} ∧ c ∧ Next ⇒ c'）后加入帧。
   - **子句推送**：当所有前沿坏状态被阻塞后，添加新帧并尝试将已准入子句向前传播；传播失败时触发一次额外阻塞尝试（最多一次LLM调用）。
   - **不动点检测**：若相邻帧等价（F_j = F_{j+1}），提取归纳不变量。
2. **与标准IC3的关键差异**：
   - 阻塞子句不是通过SAT泛化得到，而是由LLM生成，因此允许**延迟阻塞**——当前坏状态无法覆盖时，积累更多坏状态后批量提交给LLM以提供更丰富的上下文。
   - 推送阶段增加机会性阻塞，用LLM处理传播失败的反例。
3. **LLM模块**：每次提示包含完整协议规范、当前一批坏状态、从规范中提取的量词模板、已有子句历史以及验证反馈（哪些子句在可达状态上正确/错误）。LLM返回JSON数组格式的TLA+子句，经语法检查和TLC筛选后纳入子句池。

## 6. 主要结论与发现

1. **IC3Syn在所有29个协议上均成功发现候选归纳不变量**，而Endive解决24个，IC3PO解决26个，SWISS解决17个，DistAI错误最多且仅在少量协议上有效。
2. **唯一一个所有对比工具均失败的协议**——MongoLoglessDynamicRaft（MLDR），IC3Syn以4525秒成功解决，并发现15个强化子句（简化为7个），证明无界安全性。
3. **IC3架构是成功的关键**：直接LLM基线（无IC3）在6个困难协议上超时（包括MLDR和pyv-paxos-epr），而弱LLM（Gemini 3 Flash）搭配IC3仍能解决27个协议，仅超时2个（MLDR和pyv-paxos-epr）。这说明IC3结构能有效补偿LLM能力不足。
4. **LLM在不变量合成中的角色应从全局生成转为局部假设生成**：在IC3框架下，LLM只需为具体坏状态提供结构合理的阻塞子句，框架保证验证和完备性。
5. **TLAPS证明全自动**：借助LLM代理，所有29个协议的无界归纳证明可自动完成（以MLDR为例，超过3000行证明由代理生成）。

## 7. 优点

- **无需逻辑限制**：直接操作完整TLA+表达，支持集合、函数、量化，不限制于一阶可判定片段。
- **无需手动模板**：LLM自动从规范提取量词模板，无需用户构造谓词池。
- **IC3分解缓解LLM不精确**：将全局任务拆分为逐个坏状态的阻塞子任务，每个子任务简单，错误子句被验证后丢弃，不会污染不变量。
- **完备性后盾**：当LLM无法生成有用子句时，IC3控制器可回退到精确聚合阻塞，保证有限实例上的完备性（与平面CTI消除方法不同）。
- **端到端自动化**：从TLA+规范输入到TLAPS证明输出，仅需少量人工干预（主要检查证明方向）。
- **跨任务复用**：子句池使得之前学到的子句可覆盖后续坏状态，减少LLM调用次数（例如两阶段提交协议仅需2次LLM调用）。

## 8. 不足与局限

1. **有限实例验证**：IC3Syn在有限实例上验证归纳性，其不动点不保证自动推广到无界实例，需额外TLAPS证明。虽然论文中此环节由LLM代理完成，但并非框架内自动完成，可能成为瓶颈。
2. **基准覆盖有限**：29个协议主要来自公开学术集合，未涵盖更复杂的真实工程系统（如更大的参数域、更深的嵌套量化）。IC3Syn可能面临扩展性挑战。
3. **LLM依赖性**：虽然IC3缓解了LLM不精确，但在困难协议（如MLDR）上仍依赖LLM生成正确的嵌套量化子句；更换更弱模型（Gemini 3 Flash）会导致超时，表明性能边界。
4. **计算成本高**：MLDR耗时4525秒，其中验证后端Apalache查询占主导，可能无法用于快速迭代开发。对比工具（如IC3PO）在某些协议上快得多。
5. **未与最新LLM验证工具对比**：论文未与Quokka、Lemur等新兴LLM+验证工作比较，但这些工具通常面向非TLA+输入，可比性有待讨论。
6. **潜在偏差**：Endive等工具的谓词池质量未知，不公平比较可能；DistAI错误率较高可能反映配置问题。实验仅在单一硬件和API提供商上进行。
7. **可复现性**：LLM API非确定性，实验结果可能因模型版本或采样参数不同而波动。论文未报告多次运行的平均值或方差。
