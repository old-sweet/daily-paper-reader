## 摘要
大语言模型可以从自然语言规范生成硬件描述，但生成的Verilog通常存在宽度不匹配、组合环路以及不完整的case逻辑，这些缺陷能通过语法检查但在综合或流片阶段失败。我们提出CktFormalizer，这是一个通过嵌入Lean 4的依赖类型HDL重新引导LLM驱动硬件生成的框架。Lean扮演三个角色：(i) 类型检查器：依赖类型编码位宽约束、case覆盖性和无环性，将硬件缺陷转化为编译时错误，引导迭代修复；(ii) 正确性防火墙：编译后的设计在结构上无缺陷，避免后端静默失败（基线在综合和布线过程中损失20%的正确设计；CktFormalizer保留所有设计）；(iii) 证明助手：智能体对任意输入序列和参数化宽度构建机器可检查的等价性证明，超越了基于有界SMT检查的能力。在VerilogEval（156个问题）、RTLLM（50个问题）和ResBench（56个问题）上，CktFormalizer达到了与直接Verilog生成相当的仿真通过率，同时提供了显著更高的后端可实现性：95-100%的编译设计完成了完整的综合、布局布线、DRC和LVS流程。通过闭环PPA优化阶段，经过验证的架构探索实现了高达35%的面积缩减和30%的功耗降低，自动定理证明确保每个优化变体在功能上与其形式规范等价。

## Abstract
LLMs can generate hardware descriptions from natural language specifications, but the resulting Verilog often contains width mismatches, combinational loops, and incomplete case logic that pass syntax checks yet fail in synthesis or silicon. We present CktFormalizer, a framework that redirects LLM-driven hardware generation through a dependently-typed HDL embedded in Lean 4. Lean serves three roles: (i) type checker:dependent types encode bit-width constraints, case coverage, and acyclicity, turning hardware defects into compile-time errors that guide iterative repair; (ii) correctness firewall:compiled designs are structurally free of defects that cause silent backend failures (the baseline loses 20% of correct designs during synthesis and routing; CktFormalizer preserves all of them); (iii) proof assistant:the agent constructs machine-checked equivalence proofs over arbitrary input sequences and parameterized widths, beyond the reach of bounded SMT-based checking. On VerilogEval (156 problems), RTLLM (50 problems), and ResBench (56 problems), CktFormalizer achieves simulation pass rates competitive with direct Verilog generation while delivering substantially higher backend realizability: 95--100% of compiled designs complete the full synthesis, place-and-route, DRC, and LVS flow. A closed-loop PPA optimization stage yields up to 35% area reduction and 30% power reduction through validated architecture exploration, with automated theorem proof ensuring that each optimized variant remains functionally equivalent to its formal specification.

## 1. 论文的核心问题与整体含义（研究动机和背景）
- **问题**：大语言模型（LLM）能够从自然语言规范生成硬件描述（如 Verilog），但生成的代码常包含位宽不匹配、组合环路、不完整的 case 逻辑等缺陷。这些缺陷能通过语法检查，却在后端综合或流片阶段才暴露，导致设计失败，造成巨大的成本损失。
- **背景**：传统的硬件描述语言（Verilog/VHDL）缺乏静态保证（依赖类型、穷尽模式匹配、机器可检查证明）。既有工作分为两派：类型安全 HDL（如 Chisel、Bluespec）和 LLM 直接生成 Verilog，但两者尚未结合。本文提出将其结合，通过依赖类型 HDL 作为 LLM 的生成目标，使编译器成为验证器。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：将 LLM 驱动的硬件生成重定向至嵌入 Lean 4 中的依赖类型硬件领域特定语言（HDL）。Lean 同时扮演三种角色：
  - **类型检查器**：依赖类型编码位宽约束、case 覆盖和无环性，将硬件缺陷转化为编译时错误，引导智能体迭代修复。
  - **正确性防火墙**：编译通过的设计在结构上无缺陷，保证后端合成、布局布线阶段不会静默失败。
  - **证明助手**：智能体可自动构建机器检查的等价性证明（覆盖任意输入序列和参数化宽度），超越有界 SMT 检查。
- **关键技术细节**：
  - **Lean HDL 信号模型**：`Signal dom α` 表示为自然数到 α 的函数（时间步 → 值），支持周期精确仿真和形式推理。
  - **编译时保证**：位宽在类型层面编码（BitVec N）；组合逻辑构成 DAG（反馈需显式 `Signal.register`，杜绝组合环路）；穷尽模式匹配防止意外锁存。
  - **智能体迭代编译-修复循环**：LLM（Claude Sonnet 4.5）生成 Lean HDL 代码 → 调用 `lake build` 编译 → 解析类型错误 → 指导编辑 → 重复直至编译通过。系统提示包含 HDL 语法、设计模式、常见错误、工作流，并提供 10 个已验证的基准示例。
  - **SystemVerilog 提取**：通过 `#synthesizeVerilog` 元程序遍历信号图，输出可综合的 SystemVerilog（`always_ff`、`assign` 等），自动插入时钟和复位，并自动生成顶层模块包装器。
  - **PPA 优化闭环**：设计通过仿真和综合后，智能体接收 PPA 反馈（面积、功耗、最差负松弛 WNS、单元数），进行多轮架构探索（并行/串行、树形/链式规约等），每次修改后重新编译并验证等价（通过 Lean 定理证明或 Yosys SAT 检查），确保功能不变。
  - **形式等价证明**：利用信号的小延语义，可直接在 Lean 中陈述并证明 `∀ input. opt(input) = spec(input)`，涵盖所有输入和时间步。对于简单组合和时序电路，可用 `simp`、`decide` 等策略自动证明；复杂时序可借助 Yosys 的有界归纳检查作为补充。

## 8. 不足与局限
- **仿真通过率并未超越直接生成**：在某些基准（如 RTLLM）上 CktFormalizer 的 RTL 仿真通过率（40%）明显低于直接 SV（60%），主要提升在于后端物理实现。这说明依赖类型 HDL 的生成难度可能略高，智能体需要更多迭代来匹配直接生成的正确性。
- **形式证明能力有限**：对于复杂有限状态机、深度流水线或存储器重设计，Lean 的 `simp`/`decide` 策略往往不足以自动完成证明，LLM 生成的证明常需手动介入。当前主要依赖 Yosys SAT 补充，但后者仅限于有界验证。
- **不支持多时钟域和异步复位**：当前信号模型 `Signal dom α ≜ N → α` 仅支持单一同步时钟，多时钟交互和异步复位无形式语义，需依赖仿真或专门的 CDC 工具。
- **实验规模限制**：所有基准均为小规模 IP 级设计（多为几十至几千门），未验证在大型 SoC 级工程上的可扩展性和效率。
- **依赖商业 LLM**：使用 Claude Sonnet 4.5，成本高且不可控；未实验开源模型（如 Code Llama、StarCoder2），普适性有待验证。
- **PPA 优化有效性有限**：在 63 个设计中，仅 10 个实现 >5% 面积缩减，证明生成和架构探索在某些设计上仍存在失败情况。
- **消融实验不够详细**：仅在一个 50 问题子集上进行组件消融，未在全基准上全面消融。且未比较不同 LLM 主干或不同温度设置的影响。
