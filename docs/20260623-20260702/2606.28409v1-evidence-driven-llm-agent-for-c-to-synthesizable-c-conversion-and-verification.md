## 摘要
可软件编译的C程序通常无法完成高层次综合工具链的四阶段流水线——编译、C仿真、综合和C/RTL协同仿真——因为HLS只接受C的一个可综合子集（HLS-C）。然而，现有为HLS代码修复构建的大语言模型系统大多只覆盖早期流水线阶段，并将原始工具日志直接输入模型，导致修复脆弱且难以复现。我们将C到HLS-C的转换形式化为一个在HLS工具（Xilinx Vitis）上的闭环生成-验证-诊断-修复问题，贡献了三个组件：一个端到端的协作智能体工作流，由四阶段验证器在严格证据隔离下闭环；一个渐进式不匹配定位链，通过日志规范化、AST反向切片和双迹线插桩定位CSim/CoSim不匹配；以及一个基于类型化查询的两阶段证据RAG，由自演化、族路由的修复卡池支持。实验结果表明，所提出的工作流显著优于所有可比的最新模型。

## Abstract
Software-compilable C programs routinely fail to complete the four-stage pipeline of a high-level synthesis (HLS) toolchain -- compilation, C simulation (CSim), synthesis, and C/RTL co-simulation (CoSim) -- because HLS accepts only a synthesizable subset of C (HLS-C). Yet most existing large language model (LLM) systems built for HLS code repair only cover the early pipeline stages and feed raw tool logs directly to the model, yielding brittle and hard-to-reproduce fixes. We formulate C-to-HLS-C conversion as a closed-loop generation-verification-diagnosis-repair problem on an HLS tool (Xilinx Vitis), contributing three components: an end-to-end workflow of cooperating agents closed by the four-stage verifier under strict evidence isolation; a Progressive Mismatch Localization Chain (PMLC) that localizes CSim/CoSim mismatches through log normalization, AST backward slicing, and dual-trace instrumentation; and a typed-query, two-stage evidence RAG backed by a self-evolving, family-routed repair-card pool. Experimental results show that the proposed workflow substantially outperforms all comparable state-of-the-art models.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题背景**：高层次综合（HLS）工具链（如Xilinx Vitis）要求输入C/C++程序必须是其可综合子集（HLS-C），而标准C程序常包含递归、动态内存、STL容器等非可综合结构，导致无法通过HLS的四阶段流水线（编译→C仿真→综合→C/RTL协同仿真）。直接将大语言模型用于生成HDL（硬件描述语言）受限于HDL训练语料稀缺且质量不稳定。
- **现有工作局限**：现有LLM-for-HLS工作多停留在编译或C仿真阶段，未覆盖完整四阶段；遇到CSim/CoSim不匹配时直接返回原始日志给LLM，导致修复成功率低且迭代次数不可控；少量RAG系统的知识库是静态规则书，不随实验失败模式演化。
- **整体目标**：将C到HLS-C的转换形式化为一个**闭环生成–验证–诊断–修复**问题，以Vitis HLS四阶段验证器为硬约束，提出证据驱动的LLM智能体系统，实现自动、可审计的转换与验证。

## 2. 方法论：核心思想与关键技术细节

- **核心思想**：在HLS工具链上构建闭环的生成-验证-诊断-修复循环，严格分离prompt-facing证据（仅包含代码、失败阶段、错误截断、局部窗口、PMLC三层）与artifact-only痕迹（参考实现、测试标签等），避免信息泄漏；通过渐进式不匹配定位和自演化的RAG池提升修复效率。

- **关键技术细节**：
  - **闭环工作流**（4个轻量级智能体）：
    - **规划器**：从原始C生成保持接口契约的紧凑任务简报。
    - **程序员**：根据简报生成初始HLS-C候选。
    - **验证器**（Vitis HLS）：按顺序执行编译、CSim、综合、CoSim，短路机制；CoSim阶段含三重看门狗。
    - **故障分析师**：在失败时从证据层提取信息生成修复简报，程序员据此打补丁。
    - 所有历史仅保留在审计日志中，避免长上下文漂移。
  - **PM定位链**：用于处理最难的CSim/CoSim不匹配。
    - **L1**：日志归一化→输出侧证据（失败输出、首个失败周期）。
    - **L2**：AST反向切片→关键变量映射与可疑行范围。
    - **L3**：轻量级插桩+双踪校准→在原始C与HLS-C上部署同位探针，定位首个分歧周期与变量。
    - 三层证据大小由可列举预算控制，且行集单调收缩（L3 ⊆ L2 ⊆ L1 ⊆ y）。
  - **MoE式两阶段证据RAG**：
    - **阶段1**：基于故障阶段、失败家族标签、仅不匹配标志的规则分类路由到特定子池（如不匹配证据、接口/综合约束等）。
    - **阶段2**：在子池内进行结构化精确匹配（阶段、家族、符号集），可选GraphRAG跨案例链接；仅当阶段、家族和至少一个符号完全匹配时才返回提示（最多3条），否则短路返回空。
    - **自演化**：人工审计成功的修复链后（确保无泄漏）才正式加入知识池，形成“运行实验→提取证据→人工审计→丰富知识→改进检索”的数据侧循环。
  - **证据隔离**：prompt-facing证据限于简报、当前代码、失败阶段、80行错误截断、半径8的局部窗口；PMLC三层仅在不匹配阶段附加；参考实现、测试标签、人工修复、基线输出等始终在artifact-only侧，实现可审计的无泄漏。
