## 摘要
规范合成任务是指从程序实现和自然语言中自动推断出形式化规范，这对重构、转译、优化和验证至关重要，但对于大型C++代码库仍然是一个开放的挑战。现有的基于LLM的方法无法同时扩展到此类代码库，无法产生足够表达系统代码特性（如动态内存和堆分配数据结构）的规范，并且无法系统性地验证这些规范以排除不正确的候选。我们提出了Spec-Agent，一个用于在大型C++代码库中合成表达性强、经过充分验证的规范的智能体系统。Spec-Agent针对一系列规范语言：命题逻辑、一阶逻辑、命题分离逻辑和一阶分离逻辑。对于每个函数，Spec-Agent使用静态分析和运行时堆跟踪来选择适当的目标规范语言，将现有的功能测试泛化为模糊测试工具，并通过反例引导的反馈迭代地优化LLM生成的候选。我们在包含数百万行代码的开源C++代码库上评估了Spec-Agent。Spec-Agent为85%的目标函数合成了有效的规范，在模糊测试和专家验证下未观察到误报，性能优于Claude Code Opus 4.6，且token成本低10倍。

## Abstract
Specification synthesis, the task of automatically inferring formal specifications from program implementations and natural language, is important for refactoring, transpilation, optimization, and verification, yet remains an open challenge for large C++ repositories. Existing LLM-based approaches fail to simultaneously scale to such repositories, produce specifications expressive enough to capture systems-code features such as dynamic memory and heap-allocated data structures, and systematically validate those specifications to rule out incorrect candidates. We present Spec-Agent, an agentic system for synthesizing expressive, well-validated specifications across large C++ codebases. Spec-Agent targets a ladder of specification languages: propositional logic, first-order logic, propositional separation logic, and first-order separation logic. For each function, Spec-Agent uses static analysis and runtime heap tracing to select the appropriate target specification language, generalizes existing functional tests into fuzz harnesses, and iteratively refines LLM-generated candidates via counterexample-guided feedback. We evaluate Spec-Agent on open source C++ codebases comprising millions of lines of code. Spec-Agent synthesizes valid specifications for 85% of target functions, with no false positives observed under fuzzing and expert validation, outperforming Claude Code Opus 4.6 at 10x lower token cost.

## 1. 核心问题与整体含义（研究动机和背景）

- **问题**: 自动推断软件的形式化规范（specification synthesis）是实现代码重构、转译、优化和形式验证的关键步骤，但对于大型 C++ 代码库（百万行级）仍是开放挑战。
- **现有不足**: 已有 LLM 方法无法同时满足三个需求：
  1. 扩展到大型代码库；
  2. 生成足够表达力的规范，以捕获动态内存、堆分配数据结构等系统软件特性；
  3. 系统性地验证生成的规范，排除假阳性。
- **意义**: 代码合同（precondition & postcondition）是形式化文档，可辅助安全漏洞发现、遗留代码迁移（如 C/C++ 到 Rust）、以及无歧义的开发文档。

## 2. 方法论：核心思想、关键技术细节、算法流程

- **核心思想**: 结合 **分离逻辑（Separation Logic）** 和 **模糊测试（Fuzz Testing）**，构建一个端到端智能体管道。分离逻辑用于表达资源感知的内存约束；模糊测试作为强伪预言机（pseudo-oracle），利用反例反馈循环迭代优化 LLM 生成的候选规范。
- **关键技术与流程**:
  1. **代码挖掘**: 对每个目标函数 f，提取源码、文档注释、单元测试，并通过 Tree-sitter 静态分析（循环、条件、归纳变量）和运行时堆追踪（动态分析）获取特征。
  2. **模糊测试工具生成**: 用 LLM 将现有单元测试泛化为模糊测试工具（fuzz harness），提升执行路径覆盖率。
  3. **规范语言选择**: 基于特征自动选择四种逻辑中的一种：
     - 命题逻辑 → 一阶逻辑 → 命题分离逻辑 → 一阶分离逻辑（含量化器和分离合取）。
     - 选择规则：有循环 → 一阶逻辑；有堆分配 → 分离逻辑；同时需要 → 一阶分离逻辑。
  4. **规范生成**: 用 LLM 生成候选 precondition（P）和 postcondition（Q）。提示中包含目标语言语法、最多 10 个正确示例、形式文法。
  5. **规范模糊测试**: 解析候选规范，编译为运行时断言，用 libFuzzer 执行模糊测试。若发现违反则记录反例并拒绝候选。
  6. **规范精炼**: 基于反例反馈迭代 LLM 重新生成，并检查候选是否达到目标语言格（lattice）层级（Prop ⊑ FOL, Prop ⊑ Prop SL, FOL ⊑ FOSL, Prop SL ⊑ FOSL）。达到目标层级且无模糊测试错误则接受，否则继续，直到预算耗尽。

- **算法流程**（文字描述）: 输入函数 → 代码挖掘（静态+动态特征） → 生成模糊测试工具 → 选择语言等级 → LLM 生成候选 → 模糊测试验证（若违反则记录反例） → 检查语言等级是否达标 → 若不达标或未通过则重试（最多指定次数） → 最终输出有效规范。
