## 摘要
基于断言的验证（ABV）是现代硬件设计的基石，然而将设计意图手动转化为形式化的SystemVerilog断言（SVA）仍然劳动密集且容易出错。尽管大语言模型（LLM）在自动化此过程中展现出潜力，但现有基准仍受到不现实的任务设定、薄弱的规格输入和过于简化的评估的限制。为解决这些局限性，我们引入了AssertLLM2，一个面向硬件验证中真实断言生成的开源基准。AssertLLM2包含83个真实设计，涵盖13个功能类别。对于每个设计，该基准提供了结构化设计规格、经过验证的依赖完整的黄金级RTL，以及系统性突变的带有缺陷的RTL变体。这些支持两种实际场景：缺陷预防（根据规格生成断言以防止设计错误）和缺陷捕获（生成断言以揭示预期行为与错误实现之间的差异）。据我们所知，AssertLLM2是首个明确使用有缺陷的RTL作为输入来评估缺陷检测能力的基准。AssertLLM2进一步采用了更严格的评估框架，涵盖语法有效性、形式化可证明性、覆盖率和基于突变的缺陷检测。我们的基准使得对断言生成进行更真实、更广泛的评估成为可能，并为最先进的LLM在实际硬件验证中建立了严格的基线。

## Abstract
Assertion-based verification (ABV) is a cornerstone of modern hardware design, yet manually translating design intent into formal SystemVerilog Assertions (SVAs) remains labor-intensive and error-prone. While Large Language Models (LLMs) show promise for automating this process, existing benchmarks remain limited by unrealistic task formulations, weak specification inputs, and oversimplified evaluation. To address these limitations, we introduce AssertLLM2, an open-source benchmark for realistic assertion generation in hardware verification. AssertLLM2 contains 83 real-world designs across 13 functional categories. For each design, the benchmark provides a structured design specification, a verified dependency-complete golden RTL, and systematically mutated buggy RTL variants. These support two practical settings: bug-prevention, where assertions are generated from specifications to guard against design errors, and bug-hunting, where assertions are generated to expose discrepancies between intended behavior and faulty implementations. To the best of our knowledge, AssertLLM2 is the first benchmark to explicitly use buggy RTL as input to evaluate bug-detection capability. AssertLLM2 further adopts a more rigorous evaluation framework spanning syntactic validity, formal provability, coverage, and mutation-based bug detection. Our benchmark enables a more realistic and extensive assessment of assertion generation and establishes rigorous baselines for state-of-the-art LLMs in practical hardware verification.

- **研究动机**：硬件验证中，基于断言的验证（ABV）是确保设计正确性的关键。但手动将设计规格（specification）转化为形式化的 SystemVerilog 断言（SVA）既耗时又易出错，是当前验证流程的瓶颈。
- **背景**：大语言模型（LLM）在自动化生成断言方面展现出潜力，但现有基准存在三大根本缺陷：
  1. **任务设定不现实**：多数基准依赖黄金级 RTL（golden RTL）来构造断言生成任务，而实际验证中黄金 RTL 在断言生成时并不存在。
  2. **评价过于简化**：仅使用语法检查和形式化证明结果（FPV），无法区分“表面正确”与“真正有验证价值”的断言。
  3. **设计规模和规格质量有限**：现有数据集多为小模块、样本量少、规格输入混乱（如原始PDF夹杂图表），无法反映真实场景。
- **整体目标**：提出一个更现实、更全面、更严格的基准 AssertLLM2，以评估 LLM 在真实硬件验证中生成断言的能力。

## 2. 论文提出的方法论
### 核心思想
- 构建包含 **83 个真实系统级设计** 的基准，每个设计提供：
  - **结构化文本规格**（structured SPEC）：从原始文档经过 LLM 辅助处理，去除冗余、统一格式、文本化图表，经人工审核。
  - **黄金 RTL**：经过验证的完整 RTL，仅用于评估，不输入给 LLM。
  - **系统性突变的缺陷 RTL**：通过定义的 17 种 AST 级突变算子生成，每种设计 20 个单缺陷变体 + 1 个五缺陷合并变体。
- 支持两种实际验证场景：
  - **缺陷预防（bug-prevention）**：仅根据规格生成断言，用于防止设计错误。
  - **缺陷捕获（bug-hunting）**：同时输入规格和带缺陷的 RTL（buggy RTL），生成断言以暴露实现与规格的差异。
- 严格评估框架：涵盖语法、形式化证明、覆盖率（COI 覆盖率、证明覆盖率、形式覆盖率）和基于突变的缺陷检测（bug kill ratio）。

### 关键技术细节
- **结构化规格构建**：4 步 LLM 辅助流程（去冗余 → 标准化草稿 → 细化需求 → 人工验证），消除原始文档的质量差异。
- **突变缺陷生成**：基于 AST 的突变，先定义突变目标（与规格相关信号），再应用 17 种算子（如索引扰动、逻辑替换、分支删除等），经功能等价性检查（FEC）和形式化工具验证，去除等价和无效突变。
- **评估指标**：
  - 语法正确率
  - FPV 结果（Proven / CEX / Undetermined）
  - COI 覆盖率（结构连接）、证明覆盖率（实际证明需要的逻辑）、形式覆盖率（综合信息）
  - 缺陷杀伤率（killed bugs / total bugs）

## 6. 论文的主要结论与发现
1. **现有指标不足以反映实际能力**：当前 LLM 在语法和基本 FPV 上表现良好（如 Claude-Sonnet-4.5 语法成功率 93.04%，FPV 正确率 >70%），但在更严格的指标（证明覆盖率、缺陷杀伤率）上大幅下降（最佳模型缺陷杀伤率平均仅 11.69%，联合累加后 18.51%）。
2. **结构化规格有助于稳定评估**：使用结构化规格相比原始规格，消除了由文档质量引起的性能波动，使跨设计比较更可靠。
3. **缺陷捕获场景的双刃剑效应**：提供缺陷 RTL 能显著提升某些设计的缺陷检测（如 PRESENT、fft），但在复杂设计（如 openMSP430、xgate）中反而降低性能，因为 LLM 可能被错误的实现行为误导。
4. **不同模型策略差异**：GPT-5.2 倾向于生成更少但更可靠的断言（精度导向），Claude-Sonnet-4.5 更探索性（生成更多但部分薄弱），这些差异在先前基准中不可见。
5. **证明覆盖率与缺陷杀伤率弱相关**：高 COI 覆盖率不保证高证明覆盖率或高缺陷杀伤率，表明评估需要多维度。

## 7. 优点
- **任务设定更现实**：明确区分缺陷预防和缺陷捕获，并使用缺陷 RTL 作为生成输入，模拟真实验证流程。

- **数据集丰富且规模大**：83 个真实系统级设计，远超以往基准（多为几十个小模块）。
