## 摘要
高层次综合（HLS）依赖于将原始C规格转换为可综合的面向HLS的C（HLS-C）实现。原始C规格与HLS-C实现之间的功能一致性验证是HLS设计流程中一项关键但劳动密集型的任务。尽管大语言模型（LLM）最近在自动化测试平台生成方面显示出前景，但其随机性往往导致覆盖率不足、验证环境不一致以及等价性检查结果不可靠。为了解决这些局限性，我们提出了一种知识增强的、代理驱动的左移验证框架，用于在综合前自动检查原始C和HLS-C实现之间的功能一致性。该框架引入了一种双层一致性检查机制，同时强制执行配对测试平台之间的静态结构对齐和动态行为等价性，并集成符号执行和覆盖率导向的优化以提高验证完整性。此外，我们构建了一个异构的HLS验证知识图谱，为测试平台生成提供拓扑感知的推理先验，并设计了一个自主验证代理，以协调跨异构工具链的迭代优化和故障诊断。在107个HLS基准对上的实验结果表明，所提出的框架实现了0.9826的平均覆盖率和0.9533的动态一致性，优于代表性的基于AST、检索增强和迭代代理的基线方法。https://github.com/cz-5f/HLS-LeVeri.git

## Abstract
High-Level Synthesis (HLS) relies on transforming original C specifications into synthesizable HLS-oriented C (HLS-C) implementations. Functional consistency verification between original C specifications and HLS-C implementations is a critical yet labor-intensive task in HLS design flows. While Large Language Models (LLMs) have recently shown promise in automated testbench generation, their stochastic nature often leads to insufficient coverage, inconsistent verification environments, and unreliable equivalence checking results. To address these limitations, we propose a knowledge-augmented, agent-driven shift-left verification framework for automated functional consistency checking between original C and HLS-C implementations before synthesis. The framework introduces a Dual-Tier Consistency Checking mechanism that jointly enforces static structural alignment and dynamic behavioral equivalence between paired testbenches, while integrating symbolic execution and coverage-driven refinement to improve verification completeness. Furthermore, we construct a heterogeneous HLS Verification Knowledge Graph to provide topology-aware reasoning priors for testbench generation, and design an autonomous verification agent to orchestrate iterative refinement and failure diagnosis across heterogeneous toolchains. Experimental results on 107 HLS benchmark pairs demonstrate that the proposed framework achieves 0.9826 average coverage and 0.9533 dynamic consistency, outperforming representative AST-based, retrieval-augmented, and iterative agent-based baselines. https://github.com/cz-5f/HLS-LeVeri.git

## 1. 核心问题与整体含义（研究动机和背景）

- **问题背景**：高层次综合（HLS）将原始C/C++算法规格转换为可综合的HLS-C代码。由于两者在数据结构（动态内存 vs 静态内存）、库函数（标准库 vs 硬件接口）、数据类型（浮点 vs 任意精度）等方面存在显著差异，功能一致性验证成为HLS设计流程中的关键瓶颈。
- **现有局限**：传统验证依赖C/RTL协同仿真，但综合前阶段验证不足；人工构造测试平台费时且易错；即便使用LLM自动生成测试，也常因LLM的随机性导致覆盖率低、测试环境不一致、等价性检查不可靠。
- **研究意义**：提出将验证“左移”（shift-left），即在综合前自动验证原始C与HLS-C的功能一致性，可显著降低后期调试成本，提升HLS设计可靠性。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
构建一个知识增强的、自主agent驱动的闭环验证框架，将LLM、符号执行引擎KLEE、覆盖率分析工具gcov、HLS工具Vitis HLS等异构工具链集成，通过双重一致性检查和知识图谱先验知识实现自动、可靠的测试平台生成与功能等价性验证。

### 关键技术细节

1. **双层一致性检查（Dual-Tier Consistency Checking）**  
   - **静态结构对齐**：在仿真前分析测试平台对（T_B_c, T_B_h）的抽象语法树（AST），从三个维度评估一致性：
     - 输入刺激一致性（S_IO）：基于归一化Levenshtein距离衡量数据注入子树的对齐程度。
     - 控制流图一致性（S_CFG）：加权比较控制节点分布和最大嵌套深度。
     - 数据依赖图一致性（S_DDG）：通过Def-Use链集合的Jaccard相似度度量。
   - 综合静态一致性得分：\( S_{\text{static}} = \omega_1 S_{IO} + \omega_2 S_{CFG} + \omega_3 S_{DDG} \)，低于阈值 \(\tau_{\text{static}}=0.75\) 则拒绝并触发修复。
   - **动态行为等价**：在满足静态条件后，执行逐周期协同仿真，分两步断言：
     - 输入等价检查：要求每周期输入信号严格相等（\(\forall t, I_c^t \equiv I_h^t\)），否则判定为测试平台缺陷。
     - 输出等价检查：在输入对齐前提下，若输出不等（\(\exists t, O_c^t \neq O_h^t\)），则确定为设计缺陷。

2. **覆盖率驱动的测试平台构造**  
   - 集成符号执行引擎KLEE：将输入参数视为符号变量，通过路径约束求解（SMT）生成能触发罕见分支的确定性刺激向量。
   - 闭环覆盖率优化循环：使用gcov测量语句、分支、函数调用覆盖率，组合指标 \( C_{\text{total}} = \omega_{\text{stmt}} C_{\text{stmt}} + \omega_{\text{branch}} C_{\text{branch}} + \omega_{\text{call}} C_{\text{call}} \)，若低于阈值 \(\tau_{\text{cov}}=0.95\) 则提取未覆盖结构，引导LLM和KLEE补充刺激，迭代至满足条件。

3. **自主验证代理（Agent）**  
   - 形式化定义：\( \mathcal{A} = \langle \Sigma, \mathcal{T}, \mathcal{P} \rangle \)，其中\(\Sigma\)为动态状态记忆（覆盖率、一致性得分、日志），\(\mathcal{T}\)为工具链接口，\(\mathcal{P}\)为基于LLM的推理与规划引擎。
   - 故障诊断与修复策略（四个典型case）：
     - Case 1 (F_cov)：覆盖率不足 → 提取未覆盖分支的条件，用KLEE求解路径约束，LLM补充刺激。
     - Case 2 (F_static)：静态一致性不达标 → 提示LLM对齐测试平台结构。
     - Case 3 (F_dyn_in)：动态输入不匹配 → 修复测试平台生成逻辑。
     - Case 4 (F_dyn_out)：输出不匹配 → 确认为设计缺陷，报告反例。

4. **HLS验证知识图谱（Knowledge Graph）**  
   - 基于Chrysalis-HLS数据集，经数据增强和严格筛选构建1,597个高质量4元组（P_c, P_h, TB_c, TB_h）。
   - 图谱分为：覆盖率知识图谱（G_cov）和语义知识图谱（G_sem），节点包含函数、循环、指针、HLS pragma等，边表示依赖、等价、对齐关系。
   - 推理时：解析输入程序构建查询子图 → 拓扑检索（加权相似度评分） → 组装推理提示，为LLM提供结构化先验，减少搜索空间。
