# ABSTRACT 

PLCverif is the most mature open-source platform for Programmable Logic Controller (PLC) formal verification, developed at Conseil Européen pour la Recherche Nucléaire (CERN) and in production use since 2019. Yet it has two fundamental structural limitations: no support for Ladder Diagram (LD) programs – the dominant PLC notation – and reliance on Bounded Model Checking for ANSI-C Programs (CBMC) as its primary backend, which restricts verification to bounded proofs. The PLCverif authors themselves identified Efficient SMT-based Context-Bounded Model Checker (ESBMC) as the appropriate backend improvement. Prior work established ESBMC-PLC (native textual LD frontend for ESBMC with k-induction) and ESBMC-GraphPLC (graphical PLCopen XML support); together they cover LD with unbounded proofs but not Structured Text (ST) programs, and graphical LD programs with timer or counter function blocks on rung paths remain unverifiable. This paper presents ESBMC-PLC+ , a unified framework that closes the two remaining gaps: (1) a ST/Structured Control Language (SCL) frontend via the MATIEC open-source IEC 61131-3 compiler, routing C-compiled ST programs to ESBMC with nondeterministic input modeling and YAML property injection; (2) function block state semantics for graphical LD – extending the Depth-First Search (DFS) resolver to model TON /TOF /TP timers, CTU /CTD counters, and R_TRIG /F_TRIG 

edge triggers as persistent scan-cycle state variables in the GOTO Intermediate Representation (IR). ESBMC-PLC+ is the first open-source PLC verification framework to accept all three major IEC 61131-3 input formats (textual LD, graphical LD, and ST/SCL) through a single ESBMC backend providing k-induction unbounded safety proofs. A systematic feature comparison with PLCverif and an experimental evaluation on 8 benchmark programs – including programs with up to 8 integer-valued timer state variables – demonstrate that ESBMC-PLC+ matches PLCverif’s input coverage while providing strictly stronger verification guarantees. Against nuXmv’s Binary Decision Diagrams (BDD) backend (PLCverif’s unbounded prover), ESBMC-PLC+ is 400-2,000 × faster on integer-timer programs and completes proofs where nuXmv BDD times out at 120 s .

Keywords PLC, IEC 61131-3, LD, ST, PLCverif, ESBMC, BMC, k-induction, SMT, MATIEC, PLCopen XML, Industrial Control Systems, Formal Verification

# 2 Background 

2.1 PLC Architecture and the Scan Cycle 

A PLC is an industrial digital computer engineered for deterministic, real-time control of manufacturing and process automation systems. As illustrated in Figure 2-left, a PLC comprises four principal hardware components: a Central Processing Unit (CPU) that executes the user control program; non-volatile program memory that retains both the 3ESBMC-PLC+: A Unified IEC 61131-3 Formal Verification Framework as a PLCverif Successor 

program and retentive variable values across power cycles; input modules that digitise and latch physical sensor signals (pushbuttons, limit switches, thermocouples, pressure transducers) into a Process Image Input (PII) at the start of each cycle; and output modules that drive physical actuators (solenoid valves, motor contactors, indicator lamps) from a Process Image Output (PIO) written at cycle end. Execution follows a strictly deterministic, cyclic model called the scan cycle (Figure 2-right): 1. Input scan. All input module registers are read and copied atomically into the PII, freezing a consistent snapshot of the physical world for the entire scan. 2. Program execution. The user program executes sequentially – rung by rung in LD, or statement by statement in ST – reading exclusively from the PII and writing results to the PIO and internal state variables. 3. Output scan. The completed PIO is copied to the output modules, which drive the physical actuators. 4. Housekeeping. The CPU services fieldbus and Ethernet communications, updates diagnostic registers, and resets the watchdog timer to confirm a healthy scan. 5. Repeat. 

Cycle times typically range from 1 ms to 100 ms , determined by program size, I/O count, and CPU clock speed. Sensors

The cyclic execution model has a direct implication for formal verification: because the PII snapshot is taken once 

per cycle and held constant during program execution, a PLC program behaves as a synchronous reactive system – at each cycle boundary, nondeterministic inputs arrive, the program function is applied, and new outputs are produced. ESBMC-PLC+ encodes this as a while(true) loop in which input variables are re-sampled via nondet_bool() 

and nondet_int() (open-world sensor model), output coils and timer/counter state are persistent static variables (process-image semantics), and user-defined safety properties are injected as __ESBMC_assert() statements inside the loop body. The k-induction engine then proves that all assertions hold for every reachable state across all future scan counts – an unbounded safety proof over the infinite scan-cycle trajectory.

GOTO intermediate representation. Every ESBMC frontend compiles its source language into a GOTO program – a low-level, three-address control-flow graph with explicit GOTO , ASSUME , and ASSERT instructions, typed variables, and structured function calls. Once a program is in GOTO form, the entire verification backend applies uniformly regardless of the source language. This is precisely why ESBMC-PLC+ can introduce new frontends (the LD parsers, the MATIEC-based ST wrapper) without modifying the backend: they produce GOTO programs that the existing engine already processes. 

Verification algorithms. ESBMC supports four complementary proof strategies: 1. BMC. The scan-cycle loop is unrolled to bound k, yielding a loop-free formula that is discharged by an SMT solver. If satisfiable, a concrete counterexample trace is extracted. If unsatisfiable, the property holds for all executions of length ≤ k – a bounded certificate. 2. k-induction. The base case establishes that no violation occurs in the first k steps; the inductive step proves that if the property holds across any window of k consecutive steps, it must hold in the next step. Together, these two sub-goals constitute an unbounded safety proof – valid for all scan counts, not just a finite prefix. This is the algorithm used by ESBMC-PLC+ for all three frontends. 3. Property Directed Reachability (PDR) / Incremental Construction of Inductive Clauses for Indubitable Correctness (IC3). A frame-based algorithm that incrementally refines inductive invariants; effective for purely Boolean state spaces and programs with shallow proof depth. 4. Invariant inference. Abstract interpretation automatically synthesizes loop invariants that can strengthen the inductive step of k-induction, reducing the required bound k.

SMT backends. All satisfiability queries are delegated to a pluggable SMT solver. Supported solvers include Z3 [ 12 ], Bitwuzla, MathSAT, Yices, and Boolector. For PLC programs with integer timer and counter state, the bit-vector arithmetic theory ( QF_BV ) is critical: it allows ESBMC to reason about all 216 = 65 ,536 values of a 16-bit accumulator in a single symbolic query , rather than enumerating states as a BDD-based prover must. 5ESBMC-PLC+: A Unified IEC 61131-3 Formal Verification Framework as a PLCverif Successor

2.5 PLCverif 

PLCverif [ 1 , 2 ] is a formal verification framework developed at CERN and deployed in production since 2019 for safety-critical control programs of the CERN SPS-PPS accelerator complex and the GSI heavy-ion accelerator [ 2 ]. It is the most mature open-source PLC formal verification tool and the closest prior work to ESBMC-PLC+. 

Architecture. The PLCverif pipeline, shown in Figure 4, accepts programs written in Siemens SCL – a proprietary ST dialect used in TIA Portal – or, via the OpennessScripter bridge, FBD exported from TIA Portal. The frontend parses the input and produces a GOTO-like IR compatible with CBMC’s internal format. This IR is then routed to one of three backends: 1. CBMC (default): a Boolean Satisfiability (SAT)-based BMC that unrolls the scan-cycle loop to depth k and checks the resulting formula. Supports full LTL/Computation Tree Logic (CTL) temporal logic. 3. Theta : a configurable model-checking framework from Budapest University of Technology; used for experimental comparison in [2].

# 3 PLCverif: Capabilities and Structural Limitations 

We identify three structural limitations of PLCverif that together define the requirements for a successor framework. 

Limitation 1: No LD input. PLCverif’s parser targets Siemens SCL, a proprietary ST dialect. The standard open-source PLC notation – textual and graphical PLCopen XML LD – is not accepted. Users with LD programs must either (a) use a vendor tool to convert LD to ST, introducing undocumented fidelity risks [ 13 ], or (b) manually re-implement the program in SCL, which is error-prone for large safety-critical programs. 

Limitation 2: Bounded primary backend. CBMC unrolls the scan-cycle loop to a user-specified bound k and encodes the result as a SAT formula. It cannot prove that a property holds for all scan counts; it can only find violations up to bound k or confirm absence of violations within k steps. The nuXmv backend provides BDD-based unbounded proofs but is well known to scale poorly for programs with large integer or floating-point state spaces. The PLCverif authors explicitly acknowledged that ESBMC would improve CBMC’s performance [ 1]; ESBMC-PLC delivers exactly this for LD programs, and ESBMC-PLC+ extends it to ST. 

Limitation 3: Graphical format opacity. PLCverif targets Siemens TIA Portal exports, which use a proprietary XML format distinct from the open PLCopen XML standard. Graphical PLCopen XML files from OpenPLC Editor, CONTROLLINO, or Beremiz cannot be processed. Even programs converted to ST lose the structural information encoded in the graphical rung topology, which can affect the correctness of timer and counter semantics. 4.3 Shared Property Language and Scan-Cycle Encoding 

The YAML property language from ESBMC-PLC+ applies to all three input formats unchanged. The five property kinds (mutual_exclusion , invariant , absence , response , reachability ) are compiled to assert() and assume() 

statements in the GOTO IR regardless of the frontend used. The scan-cycle wrapper inserts nondeterministic input sampling at the start of each loop iteration and the assert() property checks inside the loop body, giving k-induction access to the full scan-cycle model. 

# 5 ST/SCL Frontend via MATIEC 

While the LD frontend addresses graphical, relay-logic style programs, many industrial PLC applications are imple-mented in textual languages such as ST and SFC. This chapter presents the ST/SCL frontend of ESBMC-PLC+, which extends verification support to these languages by leveraging the open-source MATIEC compiler [8]. Rather than building a custom ST parser, ESBMC-PLC+ reuses MATIEC’s code-generation infrastructure to translate ST source to C, then instruments the resulting code with a verification harness. We describe the three-step pipeline: compilation via MATIEC, scan-cycle wrapper injection, and nondeterministic input modeling. We then cover property compilation from YAML specifications and discuss the frontend’s scope and limitations, illustrated through a motor sequencing example. 

5.1 MATIEC Overview 

MATIEC [ 8 ] is an open-source IEC 61131-3 compiler maintained by the Beremiz project. Its iec2c binary accepts a ST source file and generates three C output files: • POUS.h / POUS.c : declarations and implementations of all POUs , including the scan-cycle body function for each POU. • A configuration entry file (e.g., PLC_PRG.c for a CONFIGURATION block named PLC_PRG ): the PLC configuration entry point, calling each POU’s body function in the configured task order. The filename matches the CONFIGURA-TION identifier in the source. • A resource file (e.g., Res1.c for a RESOURCE block named Res1 ): the resource definition, including the cyclic task scheduling structure. The filename matches the RESOURCE identifier in the source. MATIEC has been used in prior PLC verification work: Ukegbu and Mehrpouyan [14] compiled 40 real-world PLCopen ST programs through MATIEC/OpenPLC and verified the resulting C using CoVeriTeam (a cooperative multi-tool verification framework), confirming ESBMC’s effectiveness on MATIEC-generated code.

5.3 Property Injection for ST Programs

# 6 Graphical LD Function Block Extension 

6.1 Motivation: The ESBMC-GraphPLC Gap 

ESBMC-GraphPLC [ 6 ] documented that its DFS resolver traverses but does not represent function block nodes encountered on rung paths. For the beremiz_traffic_light program, this caused the ORANGE_LIGHT blink network – gated by a TON timer and an R_TRIG edge detector – to collapse to a constant, producing a vacuously correct SAFE 

verdict rather than a sound one. The paper excluded this program from its evaluation and identified function block state modeling as the highest-priority future work. ESBMC-PLC+ delivers that future work: the DFS resolver is extended to emit correct GOTO IR state update code for six function block types when they appear as nodes on a rung path. 

6.2 Function Block State Model 

Each function block instance in a graphical LD program corresponds to a node in the connection graph identified by its 

localId and typeName attributes. The extended resolver, when it encounters such a node during DFS, performs two actions: 1. State variable declaration : one or more static C variables are declared before the scan-cycle loop body, represent-ing the instance’s persistent state across scans. 10 ESBMC-PLC+: A Unified IEC 61131-3 Formal Verification Framework as a PLCverif Successor 

2. State update emission : update code is emitted at the point in the rung where the function block appears, updating state variables and computing the output pin value used by downstream rung elements. The function block’s output pin ( Q for timers and counters, Q for edge triggers) is then used as a boolean term in the rung energization expression, exactly as a contact variable would be.

• P1 (mutual_exclusion : Motor_A and Motor_B never simultaneously TRUE ) is proved SAFE . This state persists for exactly one scan cycle before the next scan corrects it, making it undetectable by manual inspection or simulation unless the timer fires in the exact right scan. The violation is a real safety defect: the engineer must add an explicit guard (e.g., IF NOT Motor_B AND NOT Timer.Q THEN Motor_A := TRUE ) to prevent the transition race. This benchmark demonstrates that the ST frontend not only translates programs correctly but finds subtle timing-related safety violations that escape manual review.

program (C1), which was excluded from ESBMC-GraphPLC, and verifies all three properties correctly. 

beremiz_traffic_light (C1). The program implements a pedestrian traffic-light controller with three rungs: (i) a rising-edge trigger ( R_TRIG ) that detects a button press and latches a request; (ii) an on-delay timer ( TON ) that drives the orange-light blink phase; and (iii) a coil rung that sets the green/red outputs based on the timer output. 

Before this work , this program produced a vacuous GOTO IR under ESBMC-GraphPLC. Because the DFS resolver did not model function blocks, the TON output ( Timer.Q ) and the R_TRIG output ( TRIG.Q ) were left undriven – both collapsed to their default value of false . As a consequence, ORANGE_LIGHT was always false and the green/red outputs were trivially constant. Every safety property asserted on these outputs trivially held – not because the program was safe, but because the verifier was effectively checking a degenerate stub rather than the real program. This class of error, known as vacuous verification , is particularly dangerous: the tool reports SAFE even though it has verified nothing meaningful about the actual control logic. 

With the function block extension , the program produces a full GOTO IR in which R_TRIG maintains a persistent 

prev_Q state variable and TON maintains a persistent accumulator and output-latch. The generated IR correctly models all inter-scan dependencies introduced by these function blocks. ESBMC then verifies 3 safety properties in 37 ms 

using k-induction (base case at k = 1 , inductive step at k = 2 ): • P1 : The green light for vehicle traffic and the pedestrian walk signal are never simultaneously active. • P2 : The pedestrian walk signal does not coincide with the vehicle green signal in the crossing direction. • P3 : The pedestrian signals for the two directions are mutually consistent. All three properties are proven SAFE. The result is non-vacuous: the IR contains reachable states in which the timer fires and transitions between phases. Hence, the k-induction proof covers the controller’s full operational cycle.

# 9 Discussion 

Why MATIEC rather than a native ST parser? A native ST parser inside ESBMC would provide tighter integration but requires substantial engineering effort (MATIEC itself is >30,000 lines of C++). Routing through MATIEC reuses a battle-tested, IEC-conformant compiler maintained by an active community and already validated against real PLC programs [ 14 ]. The wrapping approach adds one external dependency ( iec2c ), which is a reasonable trade-off given the coverage gain. A native ESBMC ST frontend remains a future direction. 

Function block coverage in graphical LD. The six function block types covered ( TOF /TOF /TP , CTU /CTD ,

R_TRIG /F_TRIG ) account for the majority of function blocks appearing in real graphical LD programs surveyed for ESBMC-GraphPLC. Vendor-specific blocks (Siemens MOVE_BLK , Rockwell MSG , CODESYS network blocks) are reported as unsupported and fall back to nondeterministic over-approximation, which is sound but may produce spurious 

SAFE results for properties that depend on the block’s output. 

Scan-cycle count vs. physical time. Timer presets in the GOTO IR are interpreted as scan-cycle counts, not physical milliseconds. This is the same convention as ESBMC-PLC [ 5]. Engineers must convert: a TON with PT := 10 and 18 ESBMC-PLC+: A Unified IEC 61131-3 Formal Verification Framework as a PLCverif Successor 

a 10 ms cycle period verifies a 100 ms on-delay. The ST frontend inherits MATIEC’s IEC TIME literal handling (T#100ms ), but converts to scan-count units during wrapping using a configurable cycle period parameter (default: 

10 ms ). 

The PLCverif replacement argument. ESBMC-PLC+ does not claim to replace PLCverif for Siemens-specific workflows (SCL/STL with proprietary extensions) or for properties requiring full temporal logic. It is a drop-in replacement for PLCverif in the most common industrial scenario: a standard LD or ST program with safety invariant or response properties, where the engineer wants an unbounded proof rather than a bounded check. 

# 10 Threats to Validity 

Construct validity. The function block encodings (Table 4, Listings 5–6) are designed to match Ebnenasir’s formal LD semantics [ 9] for the elements it covers, but have not been formally proved equivalent to the IEC 61131-3 standard. The ST frontend’s correctness rests on MATIEC’s IEC conformance, which has been validated against real PLC programs [14] but not formally proved. 

External validity. The ST benchmark suite (D1; extended suite from Ukegbu and Mehrpouyan [14] pending access) is drawn from open PLCopen repositories. Proprietary CERN SCL programs (e.g., SPS-PPS) cannot be redistributed for evaluation. The graphical LD function block benchmark suite is limited by the availability of real graphical programs with timer/counter rung paths in open repositories; the beremiz_traffic_light gap has been closed, but the total number of function-block-containing graphical programs evaluated remains small. 

Internal validity. Nondeterministic input modeling (both in the LD and ST frontends) assumes a single-task, open-world sensor model. Multitask programs with shared variables under preemptive scheduling are over-approximated as single-task; this is sound (no missed violations) but may produce false positives for properties that rely on task scheduling order. 

MATIEC version dependency. The ST frontend has been tested with MATIEC commit 7949c0b (https://github. com/beremiz/matiec ) on the Beremiz GitHub repository. Future MATIEC versions that change the generated C structure (e.g., the <config_name>_run__ calling convention or __LOCATED_VAR macro expansion) may require updates to the wrapping logic. 

# 11 Future Directions 

Native ST parser inside ESBMC. Eliminating the MATIEC external dependency would allow direct ST →GOTO-IR translation, reducing the trusted code base and simplifying installation. K-ST [ 13 ] provides a validated formal semantics for ST that could serve as the translation specification. 

Siemens SCL dialect support. PLCverif’s core use case is Siemens SCL, a proprietary ST dialect with Siemens-specific function blocks ( MOVE_BLK , BLKMOV , etc.). Adding SCL dialect support to the MATIEC frontend (or adding ESBMC-PLC+ as an alternative backend to PLCverif via a plugin interface) would enable ESBMC-PLC+ to process CERN and GSI programs directly. 

Full LTL/CTL via ESBMC’s temporal logic engine. ESBMC supports LTL properties via PDR/IC3 for finite-state models. Exposing this as a property language option (e.g., –ltl-spec ) would close the remaining gap with PLCverif’s nuXmv backend without requiring BDD-based model checking. 

Ladder Logic Bombs (LLB) detection. The security application of [ 16 , 17 ] maps directly to ESBMC-PLC+’s 

absence and reachability property kinds. An ESBMC-based LLB detector for both LD and ST programs would combine formal rigor with industrial relevance and could be evaluated on the SWaT LD dataset. 

Multi-POU and multitask LD. The single-POU limitation of the LD frontend can be addressed by extending the PLCopen XML parser to process multiple POU declarations and generate cross-rung dependency analysis. Multitask support [18, 19] requires partial order reduction or compositional verification. 

Formal equivalence proof via K-LD. A K-framework semantics for LD – extending K-ST [ 13 ] – would elevate the translation and function block encodings from empirically validated to formally proved, providing a machine-checked equivalence proof for the entire ESBMC-PLC+ translation pipeline. 19 ESBMC-PLC+: A Unified IEC 61131-3 Formal Verification Framework as a PLCverif Successor 

# 12 Conclusion 

This paper presented ESBMC-PLC+ , a unified framework for formal verification of IEC 61131-3 PLC programs that extends ESBMC-PLC and ESBMC-GraphPLC with two new capabilities: a Structured Text frontend via MATIEC and function block state semantics for graphical LD programs. Together with the inherited LD support, ESBMC-PLC+ is the first open-source PLC verification framework to accept all three major IEC 61131-3 input formats – textual LD, graphical LD, and ST/SCL – through a single ESBMC backend providing k-induction unbounded safety proofs. The ST frontend routes MATIEC-compiled C through ESBMC’s C frontend with nondeterministic input sampling and YAML property injection, replacing PLCverif’s CBMC backend with ESBMC’s k-induction engine. The function block extension models six standard function block types ( TON /TOF /TP , CTU /CTD , R_TRIG /F_TRIG ) as persistent scan-cycle state variables in the GOTO IR, closing the gap that excluded timer-gated graphical LD programs from ESBMC-GraphPLC’s evaluation. A systematic feature comparison with PLCverif demonstrates that ESBMC-PLC+ matches PLCverif on input language coverage (adding native LD) while exceeding it on proof strength ( k-induction vs. bounded CBMC) and arithmetic preci-sion (SMT bit-vector vs. SAT). The experimental evaluation on 18 benchmark programs confirms correct classification with zero false positives and zero regressions on all 16 inherited benchmarks (13 ESBMC-PLC, 3 ESBMC-GraphPLC) and correct results for the two new benchmarks (C1 and D1). A direct comparison against nuXmv 2.2.0’s BDD backend – PLCverif’s unbounded prover – demonstrates that ESBMC-PLC+ is 400–2,000 × faster on programs with integer timer state and successfully verifies programs where nuXmv BDD times out at 120 s .For automation engineers working with standard PLCopen XML LD or standard IEC 61131-3 ST programs and safety invariant or response properties, ESBMC-PLC+ provides a complete alternative to PLCverif that is simpler to use (YAML properties, no temporal logic expertise required) and provides strictly stronger guarantees (unbounded 

k-induction proofs for all scan counts).
