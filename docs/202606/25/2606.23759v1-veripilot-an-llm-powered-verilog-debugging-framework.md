## 摘要
Verilog调试仍然是数字电路设计中最耗时的阶段之一。近年来，大语言模型（LLM）的进步使得自动化调试成为可能；然而，现有方法大多仅以端到端的方式依赖测试输出和编译器反馈，限制了其在复杂错误上的有效性。一个关键挑战是，错误的根本原因可能远离其可观测输出，使得LLM难以追踪代码中的长依赖链。这一挑战在大型代码库中进一步加剧，因为较长的上下文长度会阻碍高效推理。为解决这些限制，我们提出了VeriPilot，一个基于大语言模型的调试框架，它利用黄金参考模型实现细粒度的错误定位与修复。VeriPilot超越了输出级比较，通过基于LLM的分析对齐Verilog设计与其对应黄金模型的内部变量语义。然后，它利用从静态分析中提取的控制-数据流图（CDFG）逐步进行信号追踪，从黄金模型中识别出一组最小的可疑代码区域及其正确对应部分。这些结构化的洞察随后被提供给LLM，以指导推理和自动代码修复。在NVIDIA的综合Verilog设计问题（CVDP）基准上的实验结果表明，VeriPilot将GPT-4o的修复成功率从54.3%提升至85.71%，显著提高了复杂Verilog设计的错误定位准确性和修复效果。源代码和基准已在GitHub上公开：https://github.com/YihanWn/VeriPilot.git。

## Abstract
Verilog debugging remains one of the most time-consuming stages in digital circuit design. Recent advances in Large Language Models (LLMs) have enabled automated debugging; however, most existing approaches rely solely on test outputs and compiler feedback in an end-to-end manner, limiting their effectiveness on complex bugs. A key challenge is that the root cause of an error may be far removed from its observable outputs, making it difficult for LLMs to trace long dependency chains in code. This challenge is further exacerbated in large codebases, where long context lengths hinder efficient reasoning. To address these limitations, we propose VeriPilot, an LLM-powered debugging framework that leverages golden reference models to enable fine-grained bug localization and repair. VeriPilot goes beyond output-level comparison by aligning internal variable semantics between the Verilog design and its corresponding golden model through LLM-based analysis. It then performs step-by-step signal tracing using Control-Data-Flow Graphs (CDFGs) derived from static analysis, identifying a minimal set of suspicious code regions along with their correct counterparts from the golden model. These structured insights are subsequently provided to the LLM to guide reasoning and automated code repair. Experimental results on the Comprehensive Verilog Design Problems (CVDP) benchmark from NVIDIA demonstrate that VeriPilot improves the repair success rate of GPT-4o from 54.3\% to 85.71\%, significantly enhancing both bug localization accuracy and repair effectiveness for complex Verilog designs. The source code and benchmark are publicly available at Github https://github.com/YihanWn/VeriPilot.git.

### **2. 论文提出的方法论：核心思想、关键技术细节**

*   **核心思想**：VeriPilot模拟了人工调试工作流，它超越了简单的输出比较，通过**对齐Verilog设计与高层黄金模型（如Python/C）的内部变量语义**，利用**控制-数据流图（CDFG）** 进行逐步信号回溯，定位出最小的**可疑代码区域**，并将这些结构化的证据（包括正确的参考代码段）提供给LLM，引导其进行精准修复。

*   **关键技术细节（四阶段多智能体流水线）**：
    1.  **仿真与验证 (Simulation & Verification)**：
        *   **任务**：搭建跨语言（Verilog + Python）协同仿真环境，生成相同的输入向量同时驱动DUT和黄金模型，监控输出。
        *   **输出**：当输出第一次不匹配时，捕获导致失败的输入向量、不匹配的输出信号和发生不匹配的时间步，存储在共享调试上下文中。

    2.  **变量映射 (Variable Mapping)**：
        *   **任务**：将Verilog DUT的内部信号与Python黄金模型的变量建立语义对齐，这是实现细粒度调试的关键。
        *   **关键技术**：
            *   **结构化表示构建**：利用静态分析工具（PyVerilog, pyslang）为DUT和黄金模型构建CDFG，提取模块层次结构、变量属性、依赖统计等信息。并依据模块层次将变量分区，减少匹配搜索空间。
            *   **多视角变量对齐**：将变量对齐视为一个**加权二分图最大匹配问题**，综合三个维度的相似度评分：
                *   **语义相似度 (Semantic, $S_{name}$)**：利用LLM分析变量名、属性和上下文。
                *   **行为相似度 (Behavioral, $S_{trace}$)**：基于仿真时变量值变化轨迹的一致性。
                *   **结构相似度 (Structural, $S_{struct}$)**：基于CDFG中变量的扇入/扇出等轻量级描述符。
            *   **最终匹配**：通过公式 $Score(p, v) = f(S_{name}, S_{trace}, S_{struct})$ 计算候选对 $(p, v)$ 的得分，然后使用**匈牙利算法**求解最大权重匹配，并通过**层次一致性约束**（Hierarchical Consistency）避免跨模块的误匹配。

    3.  **错误定位 (Bug Localization)**：
        *   **任务**：从已映射的、最早出现行为分歧的信号（状态前沿）出发，执行严格的向后回溯，构建“可疑区域”。
        *   **流程**：（如图3所示）
            *   a) **状态前沿提取**：在所有已映射的**寄存器变量**中，根据执行时间和结构依赖，找出最早发生不匹配的寄存器，称为 `state frontier`。
            *   b) **前沿与停止边界识别**：从 `state frontier` 出发，在CDFG中向后追踪其**组合逻辑**前驱。将包含不匹配或未映射信号的节点扩展为新的 `frontier`。当回溯到所有前驱信号都与黄金模型匹配、或是原始输入时停止，这些信号构成 `stop boundary`。
            *   c) **可疑区域构建**：将 `frontier` 到 `stop boundary` 之间所有访问过的节点收集起来，形成一个最小的“逻辑锥”，即**可疑区域**。如果多个区域在结构上连接则合并。

    4.  **错误修复 (Bug Repair)**：
        *   **任务**：为LLM构建结构化的修复提示。
        *   **输入**：提示包含四部分核心证据：
            1.  **前沿信号（Frontier）**：最早出现可疑的信号。
            2.  **停止边界（Stop Boundary）**：回溯的终点。
            3.  **可疑区域代码（Suspicious Code）**：DUT中定位到的可疑RTL代码片段。
            4.  **黄金模型参考（Golden Reference）**：与可疑区域语义对应的正确Python/C代码片段。
        *   **效果**：这种结构化提示显著减少了无关上下文，引导LLM聚焦于最可能出错的逻辑，并提供了正确的行为范本，从而生成更精准的修复代码。修复后，系统会重新仿真验证，若仍有错误则进入下一轮迭代。

*   **对黄金模型的依赖**：
    *   **风险**：框架的成功高度依赖于**高质量、可执行且内部变量结构上对等的黄金模型**。如失败案例所示，当黄金模型未能体现DUT的复杂层次结构（如`counter6bit`案例）时，变量映射失败，导致定位无效。这在实际应用中可能是一个严重限制，因为许多项目可能只有非形式的规格文档，而没有可对齐内部变量的黄金模型。
    *   **假设限制**：要求工程团队同时维护一个Python/C模型和一个RTL实现，这本身就需要不小的成本。框架的应用范围可能因此受到限制。
*   **修复阶段的瓶颈**：
    *   论文明确指出，即使定位成功（找到了正确的可疑区域），从多个潜在的修复方案中选出正确的那个仍然极具挑战性，尤其是涉及跨流水线、跨周期或大范围架构修改的错误。这本质上是LLM生成能力的天花板。
*   **实验覆盖的局限**：
    *   **规模有限**：核心评估（CVDP）仅包含16个复杂案例，样本量较小。
    *   **基准选择**：Strider基准相对简单和古老。虽然CVDP更新，但16个案例的规模限制了结论的统计强度。
    *   **对比范围**：未与更多更新的2025年LLM-Agent方法（如`BugGen`, `VerilogCoder`等）进行全方位对比。
*   **代码结构敏感性**：
    *   定位算法依赖于CDFG。对于扇入/扇出极高、逻辑高度互连的设计（如Galois encryption），生成的可疑区域可能会很大，从而降低了定位的精细度。
*   **Token成本**：
    *   虽然作者认为2.58倍的开销是可接受的，但对于需要大量调试迭代或处理超大型设计的情况，成本可能会迅速累积。
